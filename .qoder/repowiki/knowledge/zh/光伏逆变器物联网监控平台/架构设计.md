- **统一入口与路由分发**：`api-gateway` 作为系统唯一对外出口，基于 `routes.go` 将 `/api/v1/*` 请求按路径前缀反向代理至 `inv-api-server`（业务逻辑）或 `inv-device-server`（实时状态/控制），并在此层统一实施 JWT 验签、RBAC 权限校验及限流。
- **双模数据摄入与解耦**：设备数据通过 `mqtt-kafka-bridge`（Webhook 模式）或直接由 `inv-device-server`（Kafka 消费者模式）进入 Kafka 消息队列，实现高吞吐遥测与告警数据的异步解耦；`inv-device-server` 负责协议解析、字段映射及 Redis 实时缓存同步。
- **服务间内部契约**：`inv-device-server` 与 `inv-api-server` 之间通过受 `INTERNAL_KEY` 保护的 HTTP 接口（`/api/v1/internal/*`）进行状态同步、OTA 进度上报及元数据更新，确保核心业务逻辑集中在 API 服务，而设备服务专注于高并发接入。
- **共享基础设施与一致性**：所有后端服务共享 PostgreSQL（TimescaleDB）作为持久化层，Redis 作为分布式会话、设备在线状态及实时数据缓存层；JWT Secret 在网关、API 服务及 EMQX 间保持一致，实现跨组件的统一身份认证。
- **容器化编排与运维**：通过 `deploy/docker-compose.yml` 统一定义服务依赖（如 DB/Redis/Kafka 健康检查）、网络隔离（`inv-net`）及环境变量注入，提供一键式全栈部署能力。
