## 3. 服务模块详解

### 3.1 Flutter APP (inv_app)

**入口**: `lib/main.dart`

**BLoC 模块** (5 个):

| Bloc | 职责 |
|------|------|
| AuthBloc | 登录/注册/登出/Token 管理/自动续期 |
| StationBloc | 电站列表/汇总/详情/创建/编辑/删除 |
| DeviceBloc | 设备列表/详情/实时数据/控制/参数/绑定/分享/OTA |
| AlarmBloc | 告警列表/详情/标记已读/处理 |
| StatisticsBloc | 数据统计 (日/月/年/总计趋势) |

**底部 Tab 导航** (5 个固定 Tab):

| Tab | 说明 |
|-----|------|
| 首页 | 电站列表、今日汇总、电站卡片 |
| 统计 | 数据统计图表 |
| 告警 | 消息告警、筛选、标记已读 |
| 设备 | 设备管理 |
| 我的 | 个人中心、设置、分享、修改密码 |

**核心服务层**:

| 服务 | 职责 |
|------|------|
| ApiService | HTTP API 调用封装 (Dio, 自动注入 Token) |
| MqttService | MQTT 连接管理、订阅主题、接收实时数据 |
| StorageService | 安全存储 Token/用户信息 |
| ServiceLocator | GetIt 依赖注入注册中心 |

### 3.2 API Server (inv_api_server)

**入口**: `cmd/main.go` | **端口**: 8080 | **框架**: Gin

**启动流程**:
1. 加载 YAML 配置 (viper)
2. 初始化日志 (zap)
3. 连接 PostgreSQL (pgxpool, 连接池 50/10)
4. 连接 Redis (go-redis)
5. 初始化 JWT 实例
6. 注册 Repository → Service → Handler
7. 启动心跳检测协程 (每 30 秒检查离线设备)
8. 注册路由 → 启动 HTTP 服务
9. 优雅关闭 (SIGINT/SIGTERM)

**分层架构**:
```
Handler (HTTP 请求处理/参数校验/响应)
   ↓
Service (业务逻辑/事务协调)
   ↓
Repository (数据库操作/SQL)
   ↓
PostgreSQL / Redis
```

**Handler 列表**:

| Handler | 文件 | 职责 |
|---------|------|------|
| AuthHandler | auth_handler.go | 登录/注册/验证码/改密/个人资料 |
| StationHandler | station_handler.go | 电站 CRUD + 汇总统计 |
| DeviceHandler | device_handler.go | 设备列表/详情/绑定/控制/参数/OTA/分享 |
| AlarmHandler | alarm_handler.go | 告警列表/详情/处理/已读 |
| AdminHandler | admin_handler.go | 管理后台 (仪表盘/设备/用户/电站/日志) |

**内部接口** (Device Server → API Server):

| 接口 | 说明 |
|------|------|
| POST /internal/device-status | 同步设备在线/离线/故障状态 |
| POST /internal/device-data | 同步设备实时遥测数据 |
| POST /internal/device-cmd-status | 同步命令执行回执 |

**心跳检测机制**:
- 每 30 秒执行 `MarkStaleDevicesOffline`
- 超过 90 秒无数据更新的设备 → `status=0` (离线)
- 离线后自动同步电站状态

**降级模式**: 当 PostgreSQL 或 Redis 不可用时，启动 minimal 模式，仅提供 `/health` 端点。

### 3.3 Device Server (inv_device_server)

**入口**: `cmd/main.go` | **端口**: 8081 | **框架**: Gin

**启动流程**:
1. 加载 YAML 配置
2. 初始化日志
3. 连接 PostgreSQL (自动执行 initSchema 建表)
4. 连接 Redis
5. 创建 Hub (消息中心)
6. 连接 MQTT Broker (自动重连)
7. 启动 DataService 4 个 goroutine
8. 注册 HTTP 路由 → 启动服务

**核心模型** (6 个):

| 模型 | 说明 |
|------|------|
| DeviceData | 遥测数据 (60+ 字段: MPPT/三相/电表/温度/功率等) |
| Device | 设备基本信息 (SN/型号/额定功率/固件版本/状态) |
| DeviceCommand | 控制命令 (设备SN/命令类型/参数/请求ID) |
| CommandStatus | 命令执行回执 (状态/消息/时间戳) |
| AlarmEvent | 告警事件 (设备SN/故障码/级别/描述) |
| DeviceParams | 设备参数配置 (充放电限制/保护阈值等) |

**DataService 4 协程**:

| 协程 | 数据源 | 处理逻辑 |
|------|--------|----------|
| processData | dataChan | 写入 telemetry 表、UPSERT 实时数据表、写 Redis、同步状态给 API |
| processCmdStatus | cmdStatusChan | 转发命令回执给 API Server |
| processAlarm | alarmChan | 查设备归属、写入 alarms 表、更新设备故障状态 |
| detectOffline | ticker(30s) | 检查所有已知设备的 lastSeen, 标记离线并通知 API |

## 4. 核心数据流程（6条关键链路）

### 流程 1：设备实时数据上报（设备 → 用户 APP）— 最核心链路

```

    │                         │                       │                       │                       │                   │
    │ 每3秒采集电气数据        │                       │                       │                       │                   │

    │ 电网电压/频率            │                       │                       │                       │                   │
    │ 电池SOC/温度             │                       │                       │                       │                   │
    │ 输出功率/故障码          │                       │                       │                       │                   │
    │──── UART 0x02 ──────►   │                       │                       │                       │                   │
    │ (PUBLISH命令)           │                       │                       │                       │                   │
    │                         │ 解析UART帧(0xAA协议)   │                       │                       │                   │
    │                         │ 组装MQTT消息           │                       │                       │                   │
    │                         │ 主题: cs_inv/{SN}/ac   │                       │                       │                   │
    │                         │ QoS 0                  │                       │                       │                   │
    │                         │──── MQTT ────────────► │                       │                       │                   │
    │                         │                       │ 消息转发              │                       │                   │
    │                         │                       ├── MQTT ────────────► │                       │                   │
    │                         │                       │                      │ handleDataMessage()   │                   │
    │                         │                       │                      │ 1. 提取 SN             │                   │

### 5.4 JSONB 动态数据与字段映射机制

系统大量使用 PostgreSQL 的 **JSONB** 数据类型，实现灵活的动态数据结构，避免频繁的数据库 schema 变更。

#### 5.4.1 JSONB 使用场景

| 表 / 字段 | JSONB 用途 | 说明 |
|-----------|-----------|------|
| `device_telemetry.data` | 存储完整原始 JSON 遥测数据 | 保留设备上报的完整数据，结构可随设备固件升级而变化 |
| `device_realtime_data.mppt_voltage` | 存储 MPPT 电压数组 | 可变长度数组，不同型号 MPPT 路数不同 |
| `device_realtime_data.mppt_current` | 存储 MPPT 电流数组 | 同上 |
| `device_realtime_data.string_currents` | 存储光伏组串电流数组 | 不同电站组串数量不同 |
| `device_models.data_fields` | 设备型号的标准字段定义 | 定义每种型号有哪些字段、类型、单位、标签 |
| `device_models.field_mapping` | MQTT Topic 字段 → 标准字段的映射 | 不同 Topic 中的字段名到统一字段名的转换规则 |
| `device_models.mqtt_topics` | 该型号需要订阅的 MQTT Topic 列表 | 每种型号可能有不同的数据上报 Topic |
| `device_params.peak_valley_schedule` | 峰谷时段配置 | 动态时段定义（如高峰/平段/低谷的时间段） |
| `device_params.inner_loop_params` | 内部环路参数 | 不同设备类型的控制参数结构不同 |
| `user_operation_logs.operation_detail` | 操作详情 | 不同操作类型的参数结构不同 |

#### 5.4.3 设备型号注册机制

**`device_models` 表**是整个系统的"设备型号注册中心"，其设计思路是：

- 新增设备型号时只需在该表中插入一条记录，**无需修改任何后端代码**
- 系统根据 `data_fields` 知道该型号有哪些数据字段
- 根据 `field_mapping` 知道如何将 MQTT Topic 中不同命名的字段映射到统一字段名
- 根据 `mqtt_topics` 知道该型号需要订阅哪些 MQTT Topic

```sql
COMMENT ON TABLE device_models IS '设备型号注册表 - 新型号只需在此配置，无需改代码';
```

#### 5.4.4 JSONB 查询效率

PostgreSQL JSONB 支持 **GIN 索引**，可高效查询 JSON 内部字段:

```sql
-- 在 telemetry 表的 data 字段上建立 GIN 索引
CREATE INDEX idx_telemetry_data ON device_telemetry USING GIN (data);

-- 可以直接查询 JSON 内部字段
SELECT * FROM device_telemetry WHERE data ->> 'fault_code' = '101';
SELECT * FROM device_telemetry WHERE data @> '{"mode": "grid_tie"}';
```

### 5.5 原始 JSON 与结构化字段双写策略

`inv_device_server` 在数据持久化时采用**双写策略**：

```
MQTT 消息到达
    │
    ├── 解析 JSON → 提取结构化字段 → 写入 device_realtime_data (60+ 字段)
    │     (用于高效 SQL 查询、聚合统计、排序过滤)
    │
    └── 保留原始 JSON → 写入 device_telemetry.data (JSONB 字段)
          (用于数据溯源、未来新增字段回溯分析、设备调试)
```

**DeviceData 模型中的双重存储**:

```go
type DeviceData struct {
    // 结构化字段 (60+ 个明确字段)
    TotalActivePower     float64   `json:"total_active_power"`
    DailyPowerYields     float64   `json:"daily_power_yields"`
    // ... 其他电气参数字段

    // 原始数据 (不序列化到 JSON)
    RawPayload string    `json:"-"`      // 接收到的原始 MQTT payload
}

// 将结构化字段再次序列化为 JSON (用于 JSONB 存储)
func (d *DeviceData) RawJSON() json.RawMessage {
    b, _ := json.Marshal(d)
    return b
}
```

**三层缓存设计**:

| 层级 | 存储位置 | 内容 | 用途 |
|------|----------|------|------|
| 内存缓存 | `DataCache.lastData` | `*DeviceData` 对象 | 毫秒级访问最新数据，供 HTTP 查询 |
| 内存缓存 | `DataCache.lastRawJSON` | 原始 JSON 字符串 | 快速返回原始数据 |
| Redis 缓存 | `telemetry:latest:{SN}` | 原始 JSON (TTL 60s) | API Server 跨进程读取最新遥测 |
| PostgreSQL | `device_realtime_data` | 结构化字段 (60+ 列) | 持久化 + SQL 聚合查询 |
| PostgreSQL | `device_telemetry.data` | JSONB 原始 JSON | 数据溯源 + 灵活查询 |

### 5.6 fixJSONKeys — 非标准 JSON 修复机制

ESP8266 等嵌入式设备资源受限，部分老旧固件可能发送**非标准 JSON**（键名不带引号），如:

```json
// 设备发送的非标准格式 (键名无引号)
{total_active_power: 1500, work_state: "running", temp: 45.2}

// fixJSONKeys() 修复为标准 JSON
{"total_active_power": 1500, "work_state": "running", "temp": 45.2}
```

**`fixJSONKeys()` 实现原理** (`mqtt/client.go`):

```
逐字符扫描 JSON 字符串
    │
    ├── 遇到 "  → 进入字符串模式 (跳过内部所有字符)
    │
    ├── 遇到 : , { [ → 检查后面紧跟的是否为键名
    │     (非引号开头、非括号、由字母/数字/下划线组成)
    │
    └── 如果是键名 → 自动添加双引号包围
          key → "key"
```

这个函数处理的是 JSON 的非标准变体，确保即使 ESP8266 固件发送了格式不严谨的数据，后端也能正确解析和存储。

   ├── processCmdStatus()
   ├── processAlarm()
   ├── detectOffline()
8. 注册 HTTP 路由
9. 启动 HTTP 服务 :8081
10. 等待 SIGINT/SIGTERM → 优雅关闭
```

**API Server 启动**:

3. 连接 PostgreSQL (pgxpool 连接池)
4. 连接 Redis

6. 注册 Repository/Service/Handler

8. 注册路由
   ├── /health

   ├── /api/v1/internal/... (内部)

```

## 11. 架构设计特点

| 特点 | 说明 |
|:---|:---|
| **双通道通信** | HTTP 处理业务逻辑 (增删改查)，MQTT 处理实时数据推送，分工明确互不干扰 |
| **数据层解耦** | Device Server 只负责数据采集入库，API Server 只负责业务查询，通过内部 HTTP 同步状态，互不阻塞 |
| **协程并发** | Device Server 使用 4 个独立 goroutine + channel 管道处理数据/命令/告警/离线检测，天然高并发 |
| **内存缓存** | DataService 维护内存中的 `lastData` map，Hub 维护 `snToLastSeen`，毫秒级访问最新数据 |
| **双层离线检测** | Device Server 基于 MQTT 活跃度 (120s) + API Server 基于 DB 时间戳 (90s) 双重保障设备状态准确 |
| **Graceful Shutdown** | 两个 Go 服务都实现 SIGINT/SIGTERM 优雅关闭，确保正在处理的数据不丢失 |
| **降级模式** | API Server 在 DB/Redis 不可用时自动启动 minimal 模式，至少返回健康检查 |
| **Clean Architecture** | Handler → Service → Repository 标准三层架构，职责清晰，方便测试和维护 |
| **JWT 无状态认证** | 不依赖服务端 Session，适合移动端和分布式部署场景 |
| **自动 Schema 初始化** | Device Server 启动时自动执行 `CREATE TABLE IF NOT EXISTS`，无需手动执行 SQL |
| **设备端 FreeRTOS** | ESP8266 固件基于 FreeRTOS 多任务调度，状态机清晰 (AP配网→STA连接→MQTT运行) |
| **配网安全** | ESP8266 仅在 AP 模式开放 8888 端口配网，且监听地址固定 192.168.4.1，禁止外网访问 |
| **设备分享机制** | 支持设备共享 (view/control 权限)，一个设备可同时被多个用户查看或控制 |
| **五级角色体系** | 原厂/总代理/经销商/安装商/用户，后端按角色过滤数据范围，前端无差别展示 |
| **Modbus 同步限制** | Modbus 总线仅用于逆变器间数据同步，不同步参数设置，每台逆变器参数完全独立 |

## 13. Docker 部署架构

### 13.1 生产级服务编排 (`deploy/docker-compose.yml`)

生产环境使用 **EMQX 5.3** 作为 MQTT Broker（非 Mosquitto），支持完整的集群管理、TLS/SSL、Dashboard 运维面板。

| 服务 | 镜像 | 端口 | 依赖 | 说明 |
|------|------|------|------|------|
| postgres | postgres:15-alpine | 5432 | — | 关系数据库，自动执行 schema.sql + phase2_upgrade.sql |
| redis | redis:7-alpine | 6379 | — | 缓存，AOF 持久化，内存限制 256MB，LRU 淘汰策略 |
| emqx | emqx/emqx:5.3 | 1883,8083,8084,18083 | — | 生产级 MQTT Broker，带 Dashboard 管理面板 |
| device-server | 自构建 | 8081 | postgres, redis, emqx | 设备数据处理，Go 静态编译 |
| api-server | 自构建 | 8080 | postgres, redis | RESTful API，Go 静态编译 |
| nginx | nginx:alpine | 80, 443 | api-server, device-server | 反向代理 + SSL 终止 (仅 production profile) |

### 13.2 环境变量覆盖机制

```yaml
services:
  device-server:
    environment:
      - DB_HOST=postgres
      - DB_PASSWORD=${DB_PASSWORD:-inv_mqtt_2024}
      - MQTT_BROKER=emqx
      - REDIS_HOST=redis
      - TZ=Asia/Shanghai

  api-server:
    environment:
      - JWT_SECRET=${JWT_SECRET:-inv-mqtt-jwt-secret-2024-change-me}
      - DB_HOST=postgres
      - REDIS_HOST=redis
      - TZ=Asia/Shanghai
```

所有敏感信息（数据库密码、JWT Secret、EMQX 密码）通过 `${VAR:-default}` 语法支持 `.env` 文件覆盖，避免硬编码。

### 13.3 Docker Compose Profile 管理

```yaml
nginx:
    profiles:
      - production
```

使用 `docker compose --profile production up -d` 启动完整生产栈（含 Nginx），日常开发只需启动核心服务。

### 13.4 网络与存储

```yaml
networks:
  inv-network:
    driver: bridge

volumes:
  postgres_data:
  redis_data:
  emqx_data:
  emqx_log:
  device_logs:
  api_logs:
```

- 所有容器位于同一 `inv-network` 桥接网络，通过服务名互相访问
- 数据库和缓存的数据卷持久化，确保容器重启不丢失数据

### 13.5 Docker 镜像构建

Go 服务采用**多阶段构建**:

```dockerfile
# 构建阶段
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o server ./cmd

# 运行阶段
FROM alpine:3.19
RUN apk --no-cache add ca-certificates tzdata
COPY --from=builder /app/server /app/server
COPY config.docker.yaml /app/config.yaml
EXPOSE 8080
CMD ["/app/server", "-config", "/app/config.yaml"]
```

- 构建阶段与运行阶段分离，最终镜像仅含二进制 + Alpine 基础环境 (~15MB)
- `CGO_ENABLED=0` 确保纯静态链接

## 14. Nginx 反向代理

### 14.1 配置总览 (`deploy/nginx.conf`)

```
请求入口 :80/:443
    │
    ├── /.well-known/acme-challenge/ → Certbot ACME 验证
    │
    ├── :80 (HTTP) → 301 重定向到 HTTPS
    │
    └── :443 (HTTPS)  →  按 path 分发:
            ├── /api/        → api-server:8080     (RESTful API)
            ├── /device/     → device-server:8081  (设备内部API)
            ├── /mqtt        → emqx:8083           (MQTT WebSocket, 长连接)
            └── /             → api-server:8080     (管理后台/默认)
```

### 14.2 路由规则详解

| 路径 | 目标 | 特殊配置 |
|------|------|----------|
| `/api/` | api-server:8080 | HTTP/1.1 keepalive，30s 超时 |
| `/device/` | device-server:8081 | 路径前缀剥离，30s 超时 |
| `/mqtt` | emqx:8083 | WebSocket 升级，3600s 长连接超时 |
| `/` | api-server:8080 | 默认转发到管理后台 |

### 14.3 SSL/TLS 安全配置

```nginx
ssl_certificate     /etc/nginx/ssl/fullchain.pem;
ssl_certificate_key /etc/nginx/ssl/privkey.pem;
ssl_session_timeout 1d;
ssl_session_cache   shared:SSL:50m;
ssl_session_tickets off;

ssl_protocols TLSv1.2 TLSv1.3;
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:...;
ssl_prefer_server_ciphers off;
```

- 仅支持 TLS 1.2 和 1.3，禁用 TLS 1.0/1.1
- 使用强加密套件，优先 ECDHE 密钥交换

### 14.4 安全响应头

| 响应头 | 值 | 作用 |
|--------|-----|------|
| `X-Frame-Options` | `DENY` | 禁止 iframe 嵌入，防点击劫持 |
| `X-Content-Type-Options` | `nosniff` | 禁止 MIME 嗅探 |
| `X-XSS-Protection` | `1; mode=block` | 启用 XSS 过滤 |
| `Strict-Transport-Security` | `max-age=63072000` | HSTS，强制 HTTPS (2 年) |

### 14.5 Gzip 压缩

```nginx
gzip on;
gzip_vary on;
gzip_proxied any;
gzip_comp_level 6;
gzip_types text/plain text/css application/json application/javascript ...;
```

对 JSON API 响应和前端静态资源启用 Gzip 压缩，级别 6。

### 14.6 长连接优化

```nginx
upstream api_server {
    server api-server:8080;
    keepalive 32;
}
```

后端上游 keepalive 连接池 (32 个长连接)，MQTT WebSocket 代理设置 3600 秒超时。

## 15. 安全架构

### 15.1 传输安全

| 层级 | 措施 | 说明 |
|------|------|------|
| Web API | HTTPS (TLS 1.2/1.3) | Nginx SSL 终端，自动 HTTP→HTTPS 跳转 |
| MQTT | TCP :1883 (明文) / :8084 (TLS) | 生产环境推荐启用 TLS 加密 |
| 内部通信 | HTTP (localhost) | Device Server ↔ API Server 内部通信，不对外暴露 |

### 15.2 用户认证 (JWT)

**算法**: HS256 (HMAC-SHA256)

**令牌结构**:
```go
type Claims struct {
    UserID int64  `json:"user_id"`
    Phone  string `json:"phone"`
    Role   int    `json:"role"`
    jwt.RegisteredClaims  // ExpiresAt / IssuedAt / Issuer
}
```

**生命周期**:
| 参数 | 值 |
|------|-----|
| 有效期 | 7 天 (168h) |
| 刷新机制 | `RefreshToken()` 解析旧 Token 签发新 Token |
| 存储方式 | Flutter Secure Storage (加密存储) |
| 请求传递 | `Authorization: Bearer {token}` |

**中间件体系**:
```go
api.Use(middleware.Auth(jwtInstance))           // 标准认证 → 401
middleware.OptionalAuth(jwtInstance)            // 可选认证
middleware.RequireRole(minRole int)             // 角色权限
```

### 15.3 密码安全

| 措施 | 实现 |
|------|------|
| 哈希算法 | bcrypt + DefaultCost (成本系数 10) |
| 数据库存储 | `password_hash VARCHAR(255)` |
| API 防泄漏 | `PasswordHash` 标记 `json:"-"` 永不序列化返回 |
| 校验规则 | 6~20 位字符，前后端双重验证 |

### 15.4 MQTT 设备认证

| 参数 | 值 |
|------|-----|
| Client ID | 设备完整 SN (如 `CS_INV_A1B2C3D4E5`) |
| Username | 设备完整 SN |
| Password | SN + 设备密钥后 8 位 (出厂预置 16 字节密钥) |

**铁律**: 一台一个 SN，永不串机。Client ID 和主题硬编码为设备 SN。

### 15.5 验证码安全

| 机制 | 实现 |
|------|------|
| 验证码 | 6 位数字 |
| 有效期 | 5 分钟 (Redis TTL) |
| 发送冷却 | 60 秒 |
| 存储 | Redis Pipeline 原子操作 (set + cooldown) |
| 开发模式 | 固定验证码 "123456" |

### 15.6 客户端安全

| 措施 | 实现 |
|----------|------|
| Token 存储 | `flutter_secure_storage` (Android 加密 SP / iOS Keychain) |
| 自动登出 | Dio 拦截器捕获 401 → AuthBloc 触发登出 |
| 自动注入 | Dio 拦截器自动添加 `Authorization` 头 |

### 15.7 操作审计

所有敏感操作记录到 `user_operation_logs` 表 (JSONB):

| 字段 | 说明 |
|------|------|
| user_id | 操作人 |
| operation_type | `login/logout/device_control/param_modify/ota_upgrade` |
| operation_detail | JSONB，完整操作参数 |
| result | `success` / `failed` |
| ip_address | 客户端 IP |
