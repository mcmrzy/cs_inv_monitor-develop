## 1. 授权、范围与硬约束

### 1.1 已确认授权

- 用户已明确授权：现有旧告警事件、告警快照、并机状态、并机事件和三相历史数据可以丢弃。
- 因此本任务允许对下列对象执行受控的 `DROP + CREATE`，不要求迁移其中的历史业务数据：
  - `device_alarm_events`
  - `device_alarm_snapshots`
  - `device_parallel_state` / 历史同义表
  - `device_parallel_events`
  - `device_three_phase_3min`
- 授权不包含用户、设备、电站、型号、命令、OTA、工单、当前 `alarms` 主业务表和通用遥测历史；这些数据不得清除。

### 1.2 不可变规则

1. HTML 最终规范高于旧 Markdown、旧 MQTT 文档和当前代码；发现冲突时先修代码及派生文档，不反向修改最终规范迎合代码。
2. 所有上行消息统一使用完整 `{t,v,data}` 信封；业务代码不得跳过信封直接解释裸 `data`。
3. `t` 是设备采样时间（Unix 秒），必须原值保留并转换为 `event_time`；`received_at` 是服务端接收时间，两者不得混用。
4. 设备采样周期为 3 分钟；不创建 1 分钟聚合。三相数据每 180 秒由三相主机上报，并机稳定拓扑每 300 秒校准。
5. `data_hash` 由服务端对规范化后的 `data` 对象计算 SHA-256，不接受设备自报 hash 作为事实值。
6. 有效消息保存完整 `raw_envelope JSONB`；无法解析的消息保存原始 `raw_payload` 到 `device_ingest_errors`，禁止静默丢弃。
7. Web/App 只读取具名标准 DTO，不解析设备数组或数据库内部列。
8. 不把任何真实数据库、JWT、MQTT、内部 API 或推送凭据写入 Git、镜像层、日志或任务文档。

## 3. 统一数据契约

### 3.1 公共信封 DTO

```json
{
  "t": 1783676930,
  "v": 1,
  "data": {}
}
```

统一定义一份上行信封类型，Device Server 内部 DTO 和 API 内部 DTO 从同一 Go package 复用，或通过版本化契约测试保证完全一致：

- `t`: `int64`，大于 0，转换为 UTC `event_time`，原值另存 `event_timestamp`（若表中需要）。
- `v`: `int`，当前只接受已发布版本；未知版本进入接入错误表。
- `data`: topic 对应的强类型对象。
- `raw_envelope`: 完整信封 JSONB，不能只保存 `data`。
- `data_hash`: 对递归 key 排序、数字表达统一后的 `data` 计算 SHA-256 小写十六进制字符串。
- `received_at`: 服务端接收 UTC 时间。
- `quality_flags`: 越界、时钟异常、乱序等位标记。

### 3.2 告警 DTO

```json
{"t":1783676930,"v":1,"data":{"source":1,"code":8,"level":2,"state":1}}
```

- HTML 最终规范取值：`source` 为 0 PCS、1 BMS、2 MPPT、3 COMM；`level` 为 1 warning、2 fault；`state` 为 1 active、0 recovered。
- `(device_sn, source, code)` 关联同一告警实例。
- 幂等键：`device_sn + source + code + state + t + data_hash`。
- active 创建/激活；recovered 关闭对应 active，重复恢复幂等，不能关闭其他 source/code。

### 3.3 并机 DTO

- 只接受并机主机发送。
- `enabled`、`mode`、`count`、`total_rated_power`、`total_active_power`、`sync_state`、`machines[]` 必须按 HTML 校验。
- `machines` 按 `id` 规范排序后参与 hash；组内恰好一个 master，ID 唯一，最多 8 台；三相模式必须合法覆盖 L1/L2/L3。
- 幂等键：`master_sn + t + data_hash`。
- `enabled:false` 必须形成退出拓扑事件，并将当前状态置为 standalone/0/idle/空数组。

### 3.4 三相 DTO

```json
{
  "t": 1783676930,
  "v": 1,
  "data": {
    "voltage": [220.1,219.8,220.4],
    "current": [8.1,8.0,8.2],
    "active_power": [1750,1735,1770],
    "total_active_power": 5255,
    "line_voltage": [381.0,380.6,381.3],
    "frequency": 50.01,
    "voltage_unbalance": 0.3,
    "current_unbalance": 0.8
  }
}
```

- 四个相量数组长度必须恰好为 3，顺序固定 L1/L2/L3。
- 只接受已处于 `three_phase` 的主机；单机、单相并机、从机消息写接入错误，不进入事实表。
- 幂等键：`master_sn + t + data_hash`。
- 数据库可以抽取为扁平列供查询，但 API、Redis、Web/App DTO 仍使用数组形式，并保留原始信封。

## 9. 部署步骤

### 9.1 部署前

- 使用当前运行环境的 `deploy/.secrets/.env` 或等价 secret 注入；不得生成或替换用户现有凭据。
- 校验文件不存在 `${...}` 占位残留，输出只显示变量名和是否已设置，不显示值。
- 备份整个 PG 数据库并记录校验和；即使五张表允许丢弃，也必须保护其他业务数据。
- 导出五张目标表的结构和行数作为审计，不承诺恢复其数据。
- 保留当前镜像 tag/digest 和 compose 配置。

### 9.2 构建与迁移

```powershell
docker compose --env-file deploy/.secrets/.env -f deploy/docker-compose.yml build api-server device-server api-gateway inv-admin-frontend
docker compose --env-file deploy/.secrets/.env -f deploy/docker-compose.yml up -d postgres redis
# 使用仓库迁移入口执行到 039，并核对 schema_migrations
docker compose --env-file deploy/.secrets/.env -f deploy/docker-compose.yml up -d api-server device-server api-gateway inv-admin-frontend
```

- 实际 service 名称以 compose 文件为准，执行前用 `docker compose config --services` 核对。
- 迁移必须是独立、可观察步骤；不能在多个 API 实例启动时竞争执行 destructive migration。
- 服务启动顺序以健康检查为门槛，不使用固定 sleep 代替 readiness。

### 9.3 部署后烟测

1. 检查 PG18/Timescale 版本、迁移 039、五表字段/约束/策略。
2. 检查 API、Gateway、Device、Redis、MQTT/Kafka 健康。
3. 登录角色 0 与受限角色 1，验证菜单和权限。
4. 各发送一条 alarm active/recovered、parallel、three_phase 最终规范样例。
5. 查询数据库 raw envelope/hash/t/received_at，再通过 Gateway、Web、App 查询并比对。
6. 重复发送样例验证幂等；发送非法样例验证 ingest errors。
7. 观察 30 分钟：容器重启数 0、持续错误 0、DLQ 0、DataDropped 0、数据库连接稳定。

## 10. 回滚方案

### 10.1 应用回滚

- 立即停止新消息入口或切维护模式，避免新旧代码同时写不同结构。
- 按记录的 digest 回退 API、Device、Gateway、Web 镜像和 compose 配置。
- 不使用 `git reset --hard` 或未审计脚本操作生产数据。

### 10.2 数据库回滚

- 迁移 039 是已授权的数据破坏性重建，down 不能恢复旧五表内容。
- 若必须恢复旧应用兼容结构，优先恢复部署前 PG 备份到新的 PG18 volume，验证后切换；不要在唯一生产 volume 上反复 drop/create。
- 若保留新五表数据更重要，先导出新表，再恢复旧库；导出仅用于人工分析，不能直接灌回不兼容旧结构。
- 回滚后重新核对用户、设备、电站、型号、命令、OTA、工单和通用遥测行数。

### 10.3 回滚触发条件

- 迁移失败或非目标表行数变化。
- 三相数据出现全 0/错误时间，raw envelope/hash 丢失。
- 告警 active/recovered 无法闭环或误恢复其他告警。
- 跨租户 SN 可读取。
- 持续消息丢失、offset 异常推进、DLQ 无法重放。
- 任一核心服务 readiness 连续失败超过发布窗口约定值。
