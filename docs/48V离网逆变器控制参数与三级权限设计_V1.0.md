# 辰烁科技 48V 离网逆变器控制参数与三级权限设计 V1.0

**适用型号**：CS-I10-6k2（单相 220V、6.2kW、48V 电池系统）  
**文档性质**：产品/协议/平台联合评审稿  
**编写日期**：2026-07-15  
**关联文档**：`系统参数规范_48V离网逆变器.md`、`48V离网逆变器遥测协议与存储重设计.md`

> 本文回答“厂家、系统运维和终端用户分别需要控制什么”。文中的数值是软件设计边界或待验证建议，不是产品电气定值。最终硬上限必须由硬件、DSP、BMS、电池包规格、安规测试和认证结果共同签字确认；云端和 App 不能代替设备本地保护。

## 1. 结论

系统不应把逆变器做成一个“所有寄存器都可远程填写”的页面，而应拆成四类对象：

1. **运行命令**：开关输出、临时强充、复位等，有状态、超时和冷却时间。
2. **用户策略**：备电 SOC、能源优先级、定时充电等，可恢复、可解释、可撤销。
3. **安装配置**：电池类型/容量、AC 输入能力、并机拓扑等，只允许安装调试时修改。
4. **厂家安全定值**：过压/欠压/过流/过温阈值、校准和底层控制参数，不通过普通云端控制 API 暴露。

对本项目建议形成三个产品视图、四个技术角色：

| 产品视图 | 技术角色 | 主要职责 |
|---|---|---|
| 厂家端 | 厂家研发、厂家售后 | 定义型号能力和安全包络；诊断、授权维护、OTA |
| 系统支持端 | 平台运维、安装商 | 调试单台/电站、设置电池与 AC/发电机、策略模板、并机 |
| 用户端 | 设备所有者/被分享用户 | 日常开关、备电量、模式和时间计划；查看实际生效值 |

普通用户只接触“意图”，例如“保留 30% 备电”；安装商接触“系统参数”；厂家研发维护“不可越过的安全边界”。

## 3. 当前项目评估

### 3.1 已具备的基础

- `device_model_commands` 已能按型号描述命令、参数 schema、超时和风险等级。
- `device_commands` 已设计 task_id、ACK、执行、成功/失败/超时生命周期。
- `device_control_state` 已区分 desired 和 reported，适合展示“想要值/设备实际值/漂移”。
- 已定义功率、SOC、BMS、并机、查询和复位等基础命令。

这些结构应保留并扩展，不需要再建一套平行的命令系统。

### 3.2 必须先处理的 P0 问题

| 问题 | 当前表现 | 风险/结论 |
|---|---|---|
| 充电能力自相矛盾 | `set_charge_limit` 允许 6200W，BMS 最大充电电流为 60A | 51.2V × 60A 约为 3.07kW，固定 6200W 上限不成立 |
| 放电能力缺少损耗余量 | BMS 最大放电 120A，整机输出 6200W | 51.2V × 120A 约为 6.14kW，未计逆变损耗、线损和低电压工况，需重新核定电池/BMS/线束能力 |
| 控制权限等同查看权限 | `HasControlPermission()` 直接调用数据查看权限 | 被分享或只读用户可能获得控制权 |
| 未配置时 fail-open | 设备无型号或型号无控制项时允许任意命令 | 新型号/脏数据会绕过命令白名单 |
| 系统命令绕过型号能力 | `set_params`、`set_control`、`reset`、`restart`、`ota` 等固定白名单始终允许 | 可绕过风险等级、参数 schema 和型号支持性 |
| 后端只验证“命令存在” | 没有统一执行角色、在线、状态、BMS、互斥和动态限值校验 | 参数合法不等于当前工况允许执行 |
| App 得不到参数表单 | `/control-fields` 把 `parameter_schema` 丢弃并返回空 `control_params` | 参数命令会显示为无参按钮，无法正确生成表单 |
| App 静态配置与产品不符 | 出现 `grid_tied`、防逆流等并网设置，电池类型也向用户开放 | CS-I10-6k2 是离网机型，容易误配置或形成假功能 |
| “已发送”被当成结果 | App 在 HTTP 成功后提示命令成功/已发送 | MQTT 入队不代表设备 ACK，更不代表配置生效 |

在 P0 修正前，不建议向终端用户开放远程控制。

## 4. 权限和风险模型

### 4.1 风险等级

| 等级 | 含义 | UI 与流程 |
|---|---|---|
| R0 | 只读查询 | 无确认，可重试 |
| R1 | 可逆的日常策略 | 展示影响和生效值，允许用户执行 |
| R2 | 会中断供电或影响电池寿命 | 二次确认，仅设备所有者/安装商；记录审计 |
| R3 | 安装/拓扑/电气配置 | 仅安装商或厂家；要求停机、现场检查和前置条件 |
| R4 | 厂家安全、校准、Boot/OTA 核心操作 | 厂家签名授权；普通平台 API 不直接暴露 |

### 4.2 权限矩阵

符号：`●` 可修改，`△` 仅在授权/调试窗口修改，`○` 只读，`—` 不可见。

| 控制域 | 厂家研发 | 厂家售后/平台运维 | 安装商 | 终端用户 |
|---|:---:|:---:|:---:|:---:|
| 型号额定值和安全包络 | ● | ○ | ○ | — |
| 保护定值、采样校准、控制环参数 | ● | △ | — | — |
| 电池协议/化学体系/串数/容量 | ● | △ | ● | ○ |
| 日常 SOC 窗口、备电比例 | ● | ● | ● | ●（受限） |
| 充放电电流/功率硬上限 | ● | ● | ● | ○；仅选“温和/标准/快速”策略 |
| 能源优先级和时间计划 | ● | ● | ● | ● |
| AC 输入/发电机额定能力 | ● | ● | ● | ○ |
| AC 输出启停 | ● | ● | ● | ●（R2） |
| 输出电压/频率 | ● | △ | ●（调试期） | ○ |
| 并机角色、相位、机器 ID | ● | △ | ●（现场） | ○ |
| 故障复位 | ● | ● | ● | 仅允许恢复条件明确的 R2 故障 |
| OTA | ● | ●（发布包） | ○/预约 | ○（确认维护窗口） |
| 恢复出厂、校准、清累计量 | ● | △ | — | — |

“系统”不能是一个万能角色。建议沿用项目 RBAC，并新增细粒度 action，而不是继续用 `role == 0` 或数据归属替代控制授权。

建议权限码：

```text
device.control.basic          日常策略和普通启停
device.control.disruptive     会断电/强充/强放/复位
device.configure.battery      电池与充电曲线
device.configure.ac_input     市电/发电机输入
device.configure.parallel     并机、主从、相位
device.service.diagnostics    自检、日志、参数快照
device.service.factory        厂家安全定值/校准
ota.control                   OTA 发布和执行
```

## 5. 应控制的参数清单

### 5.1 运行与离网输出

| 建议 command/config key | 中文名称 | 角色 | 风险 | 关键联锁 |
|---|---|---|---|---|
| `ac_output_enable` | AC 输出启停 | 用户 | R2 | 无短路/过载锁定；关闭前提示负载断电 |
| `operating_mode` | 运行模式 | 用户 | R1/R2 | 只显示硬件支持的模式 |
| `output_power_limit_w` | 输出功率软上限 | 用户/安装商 | R1 | ≤ 动态可用功率；不能修改硬件过载保护值 |
| `output_voltage_profile` | 输出电压档位 | 安装商 | R3 | 预设 220/230V 等认证档位；停机修改，禁止自由输入 |
| `output_frequency_profile` | 频率档位 | 安装商 | R3 | 50/60Hz 认证档位；停机修改 |
| `eco_search_enable` | 空载节能/搜索模式 | 用户 | R1 | 对小功率/间歇负载提示可能无法唤醒 |
| `eco_search_threshold_w` | 搜索唤醒阈值 | 安装商 | R2 | 受型号范围限制 |
| `buzzer_enable` | 蜂鸣器 | 用户 | R1 | 严重故障声音不可永久关闭，可静音一段时间 |
| `temporary_mute_minutes` | 临时静音 | 用户 | R1 | 到时自动恢复 |

`operating_mode` 对离网机建议采用业务模式而非模糊的 `eco_mode`：

- `solar_first`：PV 先供负载，余电充电，不足由电池/AC 按策略补足。
- `backup_first`：优先保持电池 SOC，适合停电备电。
- `utility_first`：AC 可用时优先旁路/充电，适合电池保养或恶劣天气。
- `time_of_use`：按时间计划执行充电、放电和备电策略。
- `offgrid_only`：禁止 AC 充电/旁路（仅在硬件确实支持隔离时提供）。

必须由 ARM/DSP 明确定义每个模式的能源流优先级，云端不能只传一个 UI 名称。

### 5.2 电池与充放电

| 建议 key | 中文名称 | 角色 | 风险 | 说明 |
|---|---|---|---|---|
| `battery_profile_id` | 电池配置模板 | 安装商 | R3 | 选择经厂家验证的电池/BMS 模板 |
| `battery_chemistry` | 化学体系 | 安装商/厂家 | R3 | LiFePO4/NCM/铅酸；修改会切换整套约束 |
| `battery_series_cells` | 串数 | 厂家/安装商 | R3 | 必须与电池包一致；正常运行后锁定 |
| `battery_capacity_ah` | 容量 | 安装商 | R2 | 影响 C 倍率、电流上限和 SOC 估算 |
| `bms_protocol` | BMS 品牌/协议 | 安装商 | R3 | 通讯电池优先使用 BMS 下发限制 |
| `reserve_soc_pct` | 备电保留 SOC | 用户 | R1 | 推荐 UI 主参数，不能低于厂家/安装商下限 |
| `charge_stop_soc_pct` | 充电目标 SOC | 用户 | R1 | 受电池模板约束 |
| `soc_restart_hysteresis_pct` | SOC 恢复回差 | 安装商 | R2 | 防止阈值附近频繁切换 |
| `charge_current_limit_a` | 系统充电电流上限 | 安装商 | R2 | 最终值取多重限制的最小值 |
| `discharge_current_limit_a` | 系统放电电流上限 | 安装商 | R2 | 同上 |
| `ac_charge_current_limit_a` | AC 充电电流上限 | 安装商/用户策略 | R2 | 还受 AC 输入电流和负载占用限制 |
| `pv_charge_current_limit_a` | PV 充电电流上限 | 安装商 | R2 | 受 MPPT 和 BMS 限制 |
| `charge_voltage_v` | 充电目标电压 | 厂家/安装商 | R3 | 通讯锂电默认由 BMS 控制，用户不可改 |
| `float_voltage_v` | 浮充电压 | 厂家/安装商 | R3 | 仅对相应电池模板显示 |
| `low_voltage_cutoff_v` | 低压停机 | 厂家/安装商 | R3 | 不能低于 BMS/电池安全边界 |
| `low_voltage_restart_v` | 低压恢复 | 厂家/安装商 | R3 | 必须高于停机阈值并保留回差 |
| `temperature_compensation` | 温度补偿 | 厂家/安装商 | R3 | 主要用于铅酸；锂电由 BMS 温度策略控制 |
| `equalization_*` | 均衡充电参数 | 厂家/安装商 | R3 | 仅铅酸模板；LiFePO4/NCM 必须隐藏且禁止 |
| `force_charge` | 临时强制充电 | 运维/用户 | R2 | 必须带目标 SOC/功率与 TTL，到时自动退出 |
| `force_discharge` | 临时强制放电 | 运维/用户 | R2 | 与强充互斥，必须带最低 SOC 与 TTL |

动态充电电流必须满足：

```text
I_charge_effective = min(
  逆变器充电器额定电流,
  BMS 实时 CCL,
  电池容量 Ah × 电池模板允许 C 倍率,
  安装商设定上限,
  温度/电压/SOC 降额后的设备限值
)
```

动态放电电流同理使用 BMS DCL。动态可用输出功率还要考虑当前电池电压、效率、AC/PV/旁路状态和温度降额。设备端每个控制周期都应重新取最小值，不能只在命令下发时计算一次。

对于当前文档中的 16S LiFePO4，可把以下值作为样机验证起点，而不能直接当量产定值：

| 项目 | 建议产品默认思路 |
|---|---|
| 用户可调备电 SOC | 20%–50%，默认可从 20% 或 30% 验证 |
| 用户可调目标 SOC | 80%–100%，默认 95% 可作为寿命/容量折中候选 |
| SOC 回差 | 5%–10%，避免充放电频繁切换 |
| 日常充电电压 | 按电芯厂家/BMS 推荐值建立模板；不要把 58.4V（16×3.65V）直接作为日常默认 |
| 充放电电流 | 按电池包 Ah、持续 C 倍率、BMS、熔断器、接触器和线束共同核定 |

### 5.3 AC 输入、市电与发电机

即使产品叫“离网逆变器”，只要具备 AC IN/旁路/市电充电，就必须设计这一组参数。

| 建议 key | 名称 | 角色 | 风险 | 联锁 |
|---|---|---|---|---|
| `ac_input_mode` | 输入类型 | 安装商 | R3 | `utility` / `generator`；切换校验对应阈值模板 |
| `ac_input_current_limit_a` | AC 输入电流上限 | 安装商 | R2 | ≤ 断路器、线缆、插座和设备额定 |
| `ac_charge_enable` | AC 充电允许 | 用户 | R1 | 不改变安装商电流硬上限 |
| `ac_charge_limit_a` | AC 充电软上限 | 用户/安装商 | R1/R2 | 输入容量先供负载，余量才用于充电 |
| `bypass_enable` | 旁路允许 | 安装商/用户 | R2 | 设备拓扑和输入质量允许时才可用 |
| `transfer_mode` | 切换模式 | 安装商 | R3 | UPS/普通模式用预设，不开放任意电压频率窗口 |
| `ac_acceptance_profile` | 输入合格窗口模板 | 厂家/安装商 | R3 | 市电、弱电网、发电机分别认证 |
| `generator_auto_start_enable` | 发电机自动启停 | 安装商 | R3 | 必须配置反馈、暖机、冷却、最短运行时间 |
| `generator_start_soc_pct` | 发电机启动 SOC | 安装商 | R2 | < 停止 SOC，带回差 |
| `generator_stop_soc_pct` | 发电机停止 SOC | 安装商 | R2 | > 启动 SOC |
| `generator_quiet_hours` | 禁止启动时段 | 用户/安装商 | R1 | 严重低电量可按安装策略覆盖并告警 |
| `generator_warmup_s/cooldown_s` | 暖机/冷却 | 安装商 | R3 | 自动启停状态机必需 |

“AC 输入最大功率”不应使用当前 App 中固定 0–100000W 的输入框，应按电流配置并由额定电压换算，最大值来自型号、断路器和安装档案。

### 5.4 PV/MPPT

用户通常不需要控制 MPPT 电气边界。

| 建议 key/命令 | 角色 | 风险 | 设计 |
|---|---|---|---|
| `mppt_enable` | 安装商/运维 | R2 | 维护开关；正常用户只读 |
| `pv_charge_limit_a` | 安装商 | R2 | 用于限充，不改变 PV 过压/过流保护 |
| `pv_input_profile` | 厂家 | R4 | MPPT 电压范围、启动/停机点、扫描参数随硬件固化 |
| `mppt_rescan` | 运维 | R2 | 临时诊断命令，限频率、有执行结果 |
| `pv_insulation_test` | 厂家/安装商 | R3 | 仅硬件支持时提供，运行条件由设备校验 |

禁止云端修改：PV 绝对最大电压、硬件过流阈值、控制环补偿、保护延时。

### 5.5 时间计划与能源策略

时间计划应作为一个原子配置下发，不能分散成多条互相覆盖的开关。

每个计划段至少包含：

```json
{
  "id": "night_charge",
  "days": [1, 2, 3, 4, 5, 6, 7],
  "start": "23:00",
  "end": "06:00",
  "action": "ac_charge",
  "target_soc_pct": 90,
  "power_limit_w": 2500,
  "priority": 50,
  "enabled": true
}
```

规则：

- 使用设备电站时区，处理跨午夜计划；设备端无网络仍能执行。
- 禁止同优先级时间段重叠；冲突时采用明确优先级，而不是“后下发覆盖”。
- 支持临时覆盖并显示到期时间；到期恢复计划。
- 电池、温度、故障和 BMS 限制永远高于时间计划。
- 修改整张计划使用 revision/etag，避免并发覆盖。

### 5.6 负载保护

| 建议 key/命令 | 角色 | 风险 | 说明 |
|---|---|---|---|
| `load_shed_stage1_soc_pct` | 安装商 | R2 | 低 SOC 断开非关键负载继电器 |
| `load_shed_stage2_soc_pct` | 安装商 | R2 | 更低 SOC 进入更强保护 |
| `load_restore_soc_pct` | 安装商 | R2 | 必须高于切除阈值 |
| `critical_load_reserve_pct` | 用户 | R1 | 业务化展示为“停电保留电量” |
| `dry_contact_mode` | 安装商 | R3 | 发电机启动、告警或负载切除；选预设功能 |
| `relay_manual_override` | 运维 | R2 | 有 TTL，退出后恢复自动控制 |

若现有硬件没有可控继电器/负载输出，此域先只保留能力模型，不在 UI 显示。

### 5.7 并机与三相组网

现有命令方向正确，但必须由“多条裸命令”升级为一个安装向导/事务：

1. 扫描 CAN 成员并核对型号、硬件、固件和额定参数一致。
2. 确认所有成员 AC 输出关闭、无负载或已隔离、无故障。
3. 一次提交拓扑：模式、master SN、成员、machine_id、phase、期望 revision。
4. 逐台预配置但不启动；全部 ACK 后执行同步校验。
5. 同步成功后再允许组启动；任一失败则回滚或保持全组停机。

| 配置 | 权限 | 必须校验 |
|---|---|---|
| 单机/单相并联/三相 | 安装商 R3 | 硬件支持、同型号/兼容矩阵、全组停机 |
| master/从机、machine_id | 安装商 R3 | 唯一 master、ID 唯一连续 |
| L1/L2/L3 | 安装商 R3 | 三相各相成员规则、相序、同步状态 |
| 同步启动/停止 | 安装商/运维 R3/R2 | 只由 master 发起；全组执行结果 |
| 退出并机 | 安装商 R3 | 全组停止、配置清理一致 |

终端用户只看到组级“启动/停止”和每台状态，不得逐台更改角色或相位。

### 5.8 维护、诊断、复位和 OTA

| 命令 | 角色 | 风险 | 设计要求 |
|---|---|---|---|
| `query_config` | 用户/运维 | R0 | 返回完整 reported config + revision |
| `self_test` | 安装商/厂家 | R2 | 明确测试项和供电影响 |
| `export_diagnostic_snapshot` | 售后 | R1 | 运行状态、故障、版本、配置；脱敏后导出 |
| `fault_reset` | 用户/运维 | R2/R3 | 仅恢复条件满足且故障类型允许时执行 |
| `controlled_restart` | 运维 | R2 | 先停功率、保存状态、再重启；带冷却时间 |
| `factory_reset` | 厂家 | R4 | 现场/一次性授权，不得作为普通系统命令白名单 |
| `clear_energy_counters` | 厂家 | R4 | 不建议远程开放；强审计 |
| `ota_*` | 厂家/运维 | R4 | 签名、版本兼容、供电/SOC 检查、A/B 或回滚 |
| `calibration_*` | 厂家 | R4 | 物理工装/服务模式，云端普通 API 不展示 |

复位不能用一个含义不明的 `restart` 覆盖“故障复位、软重启、恢复出厂”。必须拆分命令并分别授权。

## 6. 固件必须拥有最终否决权

服务端和设备端都要校验，但设备端是最终安全裁决者。建议统一拒绝码：

```text
UNSUPPORTED_COMMAND
UNSUPPORTED_IN_CURRENT_MODE
DEVICE_OFFLINE
DEVICE_BUSY
REQUIRES_STOPPED
ACTIVE_FAULT
BMS_OFFLINE
BMS_LIMIT_EXCEEDED
BATTERY_PROFILE_MISMATCH
TEMPERATURE_DERATING
INVALID_RANGE
INVALID_RELATION
CONFIG_REVISION_CONFLICT
TOPOLOGY_CONFLICT
AUTHORIZATION_EXPIRED
PHYSICAL_CONFIRMATION_REQUIRED
```

每条命令的校验顺序：

1. 身份、设备归属、细粒度 permission。
2. 型号/固件是否支持命令，默认拒绝未知命令。
3. 参数类型、单位、缩放、静态范围、枚举。
4. 多参数关系和互斥关系。
5. 设备在线、时钟、当前模式、停机/故障/忙状态。
6. BMS 在线及 CCL/DCL、电压、温度、SOC 动态限制。
7. 并机组级状态和拓扑限制。
8. 风险确认、调试授权窗口、TTL 和冷却时间。
9. 创建设备独立 task_id，设备再次完整校验后执行。

设备回复必须包含 `stage`、`result_code`、`applied_args`、`reported_revision`。只有收到最终成功回复并且 reported 值一致，UI 才显示“已生效”。

## 7. 平台数据模型建议

### 7.1 扩展型号命令能力

在现有 `device_model_commands` 上增加或放入结构化 schema：

| 字段 | 用途 |
|---|---|
| `config_domain` | output/battery/ac_input/energy/parallel/service |
| `permission_code` | 细粒度 RBAC action |
| `operation_kind` | query/momentary/temporary/persistent/transaction |
| `requires_stopped` | 是否必须停机 |
| `requires_bms_online` | 是否依赖实时 BMS |
| `requires_group_master` | 是否只允许并机主机 |
| `confirmation_mode` | none/simple/type_sn/installer_token/physical |
| `ttl_seconds` | 临时命令强制失效时间 |
| `cooldown_seconds` | 复位、扫描等命令限频 |
| `precondition_schema` | 状态、故障、温度、SOC 等前置条件 |
| `ui_schema` | 分组、控件、帮助、风险提示；不可替代参数校验 schema |
| `min_firmware` | 最低支持版本 |

不要再通过 `device_model_fields.is_control` 表达命令；遥测字段和控制命令是不同实体。

### 7.2 新增电池配置模板

建议增加：

```text
battery_profiles
  profile_code, manufacturer, model, chemistry, series_cells,
  capacity_min/max, bms_protocol, charge/discharge envelope,
  voltage curve, temperature derating, lifecycle_status, version

device_battery_config
  device_sn, profile_id, capacity_ah, parallel_strings,
  installer_limits, reported_bms_identity, revision
```

模板经厂家发布后只读；修改必须新建版本。设备绑定模板时核对 BMS 上报厂家、型号、串数和容量，身份不匹配则禁止充放电参数下发。

### 7.3 时间计划与临时覆盖

```text
device_energy_schedules(device_sn, timezone, revision, enabled, periods)
device_control_overrides(device_sn, domain, value, expires_at, task_id)
```

临时强充/强放必须写 override 并带 `expires_at`，到期由设备本地和云端双重取消。

### 7.4 审计

每次配置变更至少记录：操作人、角色、来源、旧值、新值、设备上报值、型号/固件、task_id、IP/客户端、确认方式、拒绝原因和时间。R3/R4 建议保留不少于 3 年。

## 8. API 与 App/后台页面设计

### 8.1 能力 API

建议用命令能力接口替代当前会丢失 schema 的 `/control-fields`：

```text
GET  /devices/:sn/control-capabilities
GET  /devices/:sn/control-state
POST /devices/:sn/commands
GET  /devices/:sn/commands/:task_id
PUT  /devices/:sn/energy-schedule          (If-Match: revision)
POST /devices/:sn/control-overrides
DELETE /devices/:sn/control-overrides/:id
```

`control-capabilities` 必须返回：当前用户可见/可执行、参数 schema、UI schema、风险、前置条件、静态范围、动态有效范围、当前 reported 值和禁用原因。前端不硬编码范围，也不自行做 ×10 换算。

### 8.2 用户 App

建议只保留四页：

1. **运行**：输出开关、当前模式、能源流、临时静音。
2. **电池保护**：备电 SOC、目标 SOC、充电速度预设；显示 BMS 实时限制。
3. **能源计划**：时间段、AC 充电允许、临时覆盖及到期时间。
4. **设备信息**：安装配置只读、固件、实际/期望配置差异、命令记录。

移除或降权：`grid_tied`、防逆流、用户修改电池化学体系、自由填写 AC 最大功率、永久强充/强放、普通用户 restart。

### 8.3 系统支持端

单台设备页建议采用：

- **状态与能源流**：先确认当前工况。
- **用户策略**：代用户配置但保留审计来源。
- **安装配置**：电池模板、容量、AC 输入/发电机、输出制式。
- **并机向导**：组级事务，不展示逐条 MQTT 命令。
- **诊断与维护**：配置快照、自检、故障复位、OTA、命令生命周期。

高风险按钮旁必须展示当前前置条件；禁用时显示具体原因，不能只灰掉按钮。

### 8.4 厂家端

厂家端面向“型号能力发布”，不是随意改某台设备：

- 型号额定值、支持功能、命令能力和固件兼容矩阵。
- 电池模板和温度降额曲线。
- 安全包络版本及签名发布。
- 维修授权和一次性服务令牌。
- 批量 OTA 灰度、暂停、回滚和健康指标。

型号和安全模板发布后不可原地修改；通过新版本、验证样机、灰度和回滚流程生效。

## 10. 实施顺序

### P0：开放远控前

1. 核定 6.2kW、60A 充电、120A 放电、电池低压工况之间的硬件能力，形成经签字的型号安全包络。
2. 控制鉴权与数据查看鉴权分离；未知型号、空命令表、未知命令全部 fail-closed。
3. 删除“系统命令始终允许”绕过；所有可下发命令都进入型号能力和 RBAC 校验。
4. API 使用 `parameter_schema` 生成并验证具名参数；修复 `/control-fields` 丢 schema。
5. App 不再把 HTTP/MQTT 已发送当执行成功，接入 task 状态和 reported config。
6. 固件实现动态 BMS/温度/故障/模式联锁和明确拒绝码。

### P1：首版可用

1. 上线用户四页、系统支持端五页。
2. 引入电池模板、SOC 原子策略和带 TTL 的临时覆盖。
3. 完成 AC 输入/发电机配置和跨午夜时间计划。
4. 并机改成组级事务向导。
5. 完成全链路审计、命令历史、配置漂移提示。

### P2：规模化运维

1. 批量策略只对同型号/同固件/同电池模板设备开放，逐台产生 task_id。
2. OTA 灰度、自动暂停和回滚。
3. 动态降额曲线、设备健康评分和异常配置检测。
4. 厂家服务令牌和物理确认机制。

## 11. 验收用例

- 只读分享用户可以查看设备，但 `device.control.basic` 请求返回 403。
- 设备无 `model_id`、型号无命令、命令未知时全部拒绝，而不是兼容性放行。
- 用户设置 6200W 充电时，若 BMS CCL=60A，平台/设备按动态上限拒绝或钳位，并明确返回实际应用值。
- BMS 离线时，所有依赖 BMS 的充放电参数拒绝；AC 输出关闭等独立命令按故障策略处理。
- LiFePO4 模板下看不到且无法 API 调用均衡充电。
- `force_charge` 未带 TTL 被拒绝；到期后设备离线也能本地退出。
- SOC low ≥ high、发电机 start SOC ≥ stop SOC、恢复电压 ≤ 截止电压均被拒绝。
- 输出电压/频率、并机拓扑在运行状态下拒绝，并返回 `REQUIRES_STOPPED`。
- 三相组任何成员配置失败时不得部分启动。
- HTTP 202/MQTT sent 后 UI 显示“等待设备”；ACK 后显示“执行中”；reported revision 匹配后才显示“已生效”。
- R3/R4 操作都有完整旧值、新值、操作者、确认方式和执行结果审计。

## 12. 产品评审前厂家必须补齐的信息

以下信息不能靠软件团队猜测：

1. CS-I10-6k2 的真实拓扑：AC IN、旁路、发电机干接点、可控负载继电器是否存在。
2. 6.2kW 在 40–60V 各电池电压下的连续/峰值 DC 电流、效率和温度降额曲线。
3. 充电器连续/峰值电流，PV 充电和 AC 充电是否可叠加及总上限。
4. 线束、端子、熔断器、接触器和 BMS 的连续/峰值电流能力。
5. 支持的电池品牌、BMS 协议、通讯丢失后的安全降级策略。
6. LiFePO4/NCM/铅酸各模板的电压、电流、温度和均衡策略。
7. AC 输入合格窗口、切换时间、旁路额定、发电机兼容范围。
8. 并机最大台数、兼容型号/固件、单相/三相组网和失败回滚行为。
9. 故障码中哪些可自动恢复、用户复位、安装商复位或只能返厂。
10. 已取得或计划取得的安规/EMC/并机/电池认证及对应的不可调定值。

这些数据确认后，才能把本文的能力项转成最终寄存器、MQTT 命令 schema、数据库种子和页面表单。
