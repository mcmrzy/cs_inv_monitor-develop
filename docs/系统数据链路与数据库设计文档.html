第2章 五条核心数据链路

链路1：设备数据上报链路
设备（MQTT, cs_inv/{sn}/heartbeat）
EMQX Broker（TLS 8883）
mqtt-kafka-bridge（Webhook POST :18088）
Kafka inv-telemetry Topic
inv-device-server（有序Kafka消费者 runOrderedKafkaConsumer）
协议解析（Heartbeat V1: ac[8]/bat[23]/pv[7]/sys[11]/eng[12]/cells[2]）
SHA256 data_hash 去重 + 时钟守卫（±5min）
HTTP内部调用 inv-api-server
PostgreSQL 写入 device_telemetry_3min
触发器自动维护 device_latest_state + device_energy_day
Redis缓存更新（realtime:latest:{sn}）
链路2：API请求链路
前端/APP
api-gateway:8888（JWT+RBAC+限流）
反向代理 → inv-api-server:8080
PostgreSQL 查询 / Redis 缓存
  • 三级路由：public（无认证）、user（JWT+RBAC）、admin（JWT+RBAC+role≤1）
  • 限流策略：login 10QPS、send-code 2QPS
  • RBAC Redis缓存 TTL：300s
链路3：控制命令下发链路
客户端
api-gateway
inv-api-server
HTTP调用 inv-device-server
MQTT Publish cs_inv/{sn}/cmd
设备执行 → cs_inv/{sn}/cmd_result 上报
  • 离线命令队列：Redis List device:cmd:queue:{sn}
  • 设备上线：FlushPendingCommands，命令过期 300s
链路4：告警数据链路
设备 MQTT cs_inv/{sn}/data/alarm
EMQX → mqtt-kafka-bridge → Kafka inv-alerts
AlertConsumer（Redis MD5去重 TTL 60s）
HTTP POST /api/v1/internal/device-alarm
inv-api-server 同事务：先追加 device_alarm_events 不可变事件，再更新 alarms 当前生命周期/人工处理投影（不是旧新表兼容双写）
链路5：OTA升级链路
管理后台上传固件
inv-api-server → firmware_versions表
创建升级任务 → upgrade_tasks + device_upgrades
MQTT下发 cs_inv/{sn}/ota
设备执行 → 上报ACK/进度 → 状态同步
