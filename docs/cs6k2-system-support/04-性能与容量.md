# CS6K2 系统支持端性能、SLO 与容量规划

## 1. 规划口径

性能分为三个阶段，不得用单机小流量结果直接宣称“支持十万台”：

| 阶段 | 在线设备 | 用途 |
|---|---:|---|
| P1 | 10,000 | V1 生产验收目标 |
| P2 | 30,000 | 水平扩展和容量复核 |
| P3 | 100,000 | 架构规划/专项压测目标 |

CS6K2 设备可每 5 秒产生实时数据，但平台长期标准明细按 3 分钟落库。若 10 万台都每 5 秒入站，则平均约 20,000 条设备消息/秒；若合并为单 heartbeat，则按一条计算，分 topic 时需乘主题数。容量测试必须明确固件实际消息模型和 payload 字节数。

## 2. 服务等级目标

| ID | 指标 | P1 目标 | 测量点 |

| PER-SLO-005 | 24h 曲线 p95 | ≤1s | 单设备、正确聚合粒度 |
| PER-SLO-006 | 7d 曲线 p95 | ≤2s | 单设备小时聚合 |

## 3. 负载模型

### 3.1 设备侧

测试至少包含：

- 80% 正常在线：稳定 heartbeat。
- 10% 弱网：重复、乱序、延迟和离线补传。
- 5% 告警活跃：告警发生/恢复、快照。
- 3% 并机/三相：组级和成员数据。
- 2% OTA/控制：低频但高可靠 QoS1。
- 集中重连风暴：30% 设备在 2 分钟内重连并上报 info/status。

### 3.2 用户侧

- 设备/电站列表 40%。
- 仪表盘/最新态 25%。
- 历史曲线 20%。
- 告警/工单 10%。
- 控制/OTA/管理 5%。

读写比例、缓存命中率、租户大小和时间跨度必须记录到压测报告。

## 4. 容量计算

### 4.1 明细行数

3 分钟粒度每设备每天 480 行：

```text
rows/day = active_devices × 480
10,000 台 = 4,800,000 行/天
100,000 台 = 48,000,000 行/天
100,000 台 × 90 天 = 43.2 亿行
```

因此 90 天是业务上限，不是任何规模都无条件保证。每月按实际单行大小、索引、WAL、压缩率和剩余磁盘重新计算安全保留期。

### 4.2 存储水位

| 水位 | 动作 |
|---|---|
| 60% | 预测 30/60/90 天容量，复核压缩和异常增长 |
| 70% | 扩容计划和冷数据策略进入变更排期 |
| 80% | 停止延长明细保留；限制大导出；提高告警级别 |
| 90% | 进入容量事件响应，保护写入和审计优先级 |

告警、控制、OTA 和审计不能因普通遥测容量策略被提前删除。

## 5. 组件性能要求

### 5.1 Device Server / 消息流

| ID | 要求 |
|---|---|
| PER-ING-001 | 以 SN 作为 Kafka partition/message key；同 SN 同流有序 |
| PER-ING-002 | 批量写 PostgreSQL，批大小和 flush interval 可配置且有指标 |
| PER-ING-003 | 重试采用指数退避和上限，永久错误进入 DLQ，不阻塞整个分区 |
| PER-ING-004 | payload、队列、批次和并发有硬上限，防止内存无界增长 |
| PER-ING-005 | 单设备异常突发的 advisory lock 不影响其他设备/数据流 |

### 5.2 数据库

- 查询必须命中 `(device_sn, event_time)` 或对应业务索引。
- 大跨度查询只用连续聚合；禁止 JSONB 全表扫描和应用内拼接全量历史。
- 连接池总和不得超过数据库安全连接数的 70%，为迁移/运维保留连接。
- Timescale chunk、压缩、retention、聚合 refresh 在实际 PG18/Timescale 版本验证。
- 迁移和索引创建评估锁、WAL、磁盘双写和回滚成本。

### 5.3 API / Gateway

- 按归一化路由限流，并增加用户/租户/IP/设备对象维度，避免单账号拖垮系统。
- 列表最大 page_size=100；导出不得同步返回百万行。
- 历史响应限制最大点数（建议 1,500–3,000），超过时提高粒度。
- 避免 N+1 设备/权限/型号查询；批量加载并压测大租户。

### 5.4 Web/App

- 首屏只取必要数据；次要页签按需加载。
- 图表数据降采样，不在主线程处理超大数组。
- MQTT/WebSocket 重连指数退避加随机抖动；后台/不可见页面暂停非必要订阅。
- 列表虚拟化或服务端分页，避免渲染大量设备。

## 6. 降级策略

| 压力/故障 | 保留 | 降级 |
|---|---|---|
| API 高负载 | 登录、告警、控制状态、关键设备详情 | 报表、批量导出、地图、低优先级统计 |
| DB 写入压力 | 告警、命令回复、config、最新态、审计 | 普通遥测批量延迟，不丢失无记录 |
| Kafka backlog | 告警/命令/config 优先消费 | heartbeat 可延迟，显示数据年龄 |
| Redis 故障 | 权限正确、数据库事实 | 实时推送和缓存性能降低 |
| 存储高水位 | 事件/控制/审计 | 缩短明细保留，聚合继续 |

任何降级都必须产生告警和状态，不得静默返回旧数据。

## 7. 压测矩阵

| ID | 场景 | 规模/持续时间 | 通过条件 |
|---|---|---|---|
| T-PER-001 | API 混合负载 | 目标并发，30 分钟 | SLO 达标，错误率<0.1% |
| T-PER-002 | MQTT 稳态 | P1 设备，2 小时 | 无持续 backlog，内存稳定 |
| T-PER-003 | 重连风暴 | 30% P1/2分钟 | Broker/服务恢复，无认证雪崩 |
| T-PER-004 | 重复/乱序 | 10% 流量 | 幂等正确，latest 不倒退 |
| T-PER-005 | 告警风暴 | 1,000事件/s 10分钟（再按实测调整） | 不漏生命周期，通知可控 |
| T-PER-006 | 控制并发 | 1,000 设备批量 | 每台 task，权限/结果独立 |
| T-PER-007 | 90天查询 | P1 数据量 | 查询命中聚合和 SLO |
| T-PER-008 | 压缩/聚合并发 | 后台 job + 正常流量 | API/写入无不可接受尖峰 |
| T-PER-009 | DB/Redis/Kafka 故障注入 | 逐组件 5–15分钟 | 按故障矩阵降级并恢复 |
| T-PER-010 | 24小时 soak | 70% 目标负载 | 无内存/连接/队列持续增长 |
