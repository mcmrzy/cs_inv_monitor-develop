# CS6K2 系统支持端代码审查与测试计划

## 1. 目的

本计划基于当前仓库，不把已有测试数量当作质量结论。目标是通过可复现证据回答：

1. 当前代码是否符合 CS6K2 功能、协议和安全要求。
2. 正常、异常、重复、乱序、弱网、故障和恢复是否闭环。
3. 版本是否满足上线门禁；若不满足，阻断项具体在哪里。

## 4. 代码审查清单

### 4.1 通用

- 变更关联需求 ID、测试 ID、协议/迁移影响和回滚说明。
- 不覆盖用户/其他分支的无关改动；无调试文件、生成缓存或秘密。
- 错误被处理并有稳定 code；无静默 catch/continue 掩盖关键失败。
- 时间统一使用 UTC 存储，显示按电站时区；event_time/received_at 不混淆。
- 数值单位、符号、缩放、边界、NaN/Inf/null/0 处理一致。
- 日志、metrics 标签基数受控，不输出密码/token/完整私人数据。

### 4.2 API Gateway

- 路由是否落在正确 public/auth/admin/permission 组。
- Gateway 与 API 权限是否一致；只有 role 0 全局旁路。
- 限流是否按归一化路由，是否可被伪造 IP/不同路径绕过。
- CORS、CSP、安全头、body size、超时和 proxy 错误是否正确。
- `/metrics` 是否应公开；生产建议仅内部网络/认证访问。

### 4.3 API Server

- handler 是否只做传输和鉴权，业务规则在 service/use-case。
- 每个 SN/station/export/control 是否做对象级授权。
- 控制是否 capability/schema/动态联锁/fail-closed；task/desired 是否一致。
- SQL 是否参数化、分页、命中索引；事务隔离和并发 revision 是否正确。
- 告警事件与投影、人工状态是否分离；重复/乱序是否确定。
- OTA 包、任务、灰度、签名、版本和审计是否闭环。

### 4.4 Device Server / Bridge

- topic SN、payload SN、认证身份是否一致。
- released schema、数组长度、类型、范围、状态相关校验是否完整。
- data_hash、同设备顺序、重试、DLQ、permanent error 分类是否正确。
- 旧样本是否只写历史；latest 更新有事件时间条件。
- 命令回复和 OTA 状态是否幂等、终态不倒退。
- 主通道和降级通道是否会双消费；Kafka key 是否为 SN。

### 4.5 Web / App

- capability/DTO 驱动，不硬编码 CS6K2 范围、数组下标和假功能。
- loading/empty/error/forbidden/stale/partial 状态齐全。
- HTTP/MQTT sent 不提前显示 success；task/reported 状态处理乱序。
- token 存储、XSS、不可信设备字符串、深链路和路由守卫安全。
- 请求/订阅取消、重连退避、页面卸载、重复 listener 和大列表性能。
- 中英文、单位、时区、小数、无障碍和响应式一致。

### 4.6 数据库和迁移

- up/down 或前向补偿方案、空库/升级库测试、事务和锁影响。
- PK/UK/FK/check/index 与幂等键、查询模式一致。
- Timescale chunk/压缩/retention/continuous aggregate 实际可创建。
- 迁移不修改已发布文件；删除前有兼容窗口和备份。
- 权限表默认角色不造成越权或全 403。

### 4.7 部署与安全

- Docker 非 root、健康检查、资源限制、日志轮换和只读配置。
- 生产不暴露 DB/Redis/API 内部端口；TLS 验证开启。
- 镜像固定 digest，CI secret 最小权限，迁移失败阻断 readiness。
- 凭据来自 secret 管理；代码、测试、文档和日志无真实值。

## 6. 自动化命令基线


### 6.2 单元

```powershell

npm run test:run                              # React/Vitest

```

### 6.3 集成

```powershell
docker compose -f deploy/docker-compose.test.yml up -d --wait
```

当前 `docker-compose.test.yml` 主要启动 PG/Redis/EMQX，未完整启动 Gateway/API/Device Server。应补成完整测试栈或由 CI 单独启动应用；必需 E2E 用例检测到服务不可达时必须 `Fatal`，不得 `Skip`。测试结束应在隔离 CI 环境执行 `down -v`，本地需显式确认环境名称，避免清理错误卷。

## 7. 必测场景

### 7.1 协议/遥测

- 正常 heartbeat/info/status/cells/parallel/three_phase。
- 缺字段、错类型、短/长数组、越界、NaN/Inf、未知版本、非法 SN。
- QoS 重投、相同 t 不同内容、乱序、时钟为 0/未来、离线补传。
- 运行/停机/故障状态相关 AC 范围；电池功率方向和能量守恒。
- DB/Redis/API/Web/App 的字段、单位、质量和时间一致。

### 7.2 权限/控制

- 全角色×设备归属×permission×命令风险矩阵。
- 未绑定型号、命令表为空、命令禁用、固件过低、未知命令全部拒绝。
- 参数 min/max、step、enum、low<high、回差、互斥、TTL、revision。
- BMS 离线/CCL/DCL、SOC、温度、故障、设备忙、必须停机和并机从机。
- sent/ACK/executing/success/failed/timeout/cancel/迟到回复/重复 task。
- reported 一致、漂移、查询恢复和并发配置冲突。

### 7.3 告警/并机/OTA

- active 重复、恢复、乱序、快照缺失、人工确认/处理不被恢复覆盖。
- 并机成员不属于电站、重复 ID、无/多 master、相位错误、成员失败和回滚。
- OTA 错签名/hash/型号/target/降级、低 SOC、断网、掉电、重启、健康失败、回滚。

### 7.4 UI

- 角色页面/按钮可见性与直接 API 结果一致。
- 所有异步状态、错误码、数据过期、部分数据、配置漂移。
- 桌面/窄屏、中文/英文、深浅色、200% 字体、键盘/读屏。
- 真实 capability 生成表单，LiFePO4 不出现均衡，离网机不出现未支持并网项。

## 8. 硬件在环（HIL）计划

### 8.1 测试台

- CS6K2 量产等效样机，记录硬件/ARM/DSP/ESP/BMS 版本。
- 可编程 DC 电源或经批准电池模拟器、真实兼容电池/BMS。
- PV 模拟器或安全可控 PV 源、AC 输入/发电机模拟源。
- 可编程 AC 负载、示波器/功率分析仪、温度采集、CAN/RS485 抓包。
- 单机、两台单相并联、三台三相组网台架。
- 急停、隔离、熔断、消防和安全操作规程。

### 8.2 HIL 阻断用例

| ID | 用例 | 通过条件 |
|---|---|---|
| T-HIL-001 | 40–60V/温度范围输出降额 | 不超过签字包络，reported/告警一致 |
| T-HIL-002 | BMS CCL/DCL 动态降低 | 设备立即采用更小限制，云端不能覆盖 |
| T-HIL-003 | BMS 断连/恢复 | 按安全策略停充放/降级，恢复有回差 |
| T-HIL-004 | 过载/短路/过温/欠压 | 本地保护优先，远程复位受条件限制 |
| T-HIL-005 | 强充/强放 TTL 和云端断网 | 到期本地退出，互斥有效 |
| T-HIL-006 | AC 输入波动/切换/旁路 | 模式和认证窗口内行为正确 |
| T-HIL-007 | 设备掉电/重启中命令重投 | task 幂等，不重复副作用 |
| T-HIL-008 | 单相并联/三相组网失败 | 不部分启动，相位/同步/回滚正确 |
| T-HIL-009 | OTA 四 target 中断/错误包 | 验签、恢复/回滚、版本上报正确 |
| T-HIL-010 | 24h 额定/典型负载 soak | 无保护误触发、温升/性能符合规格 |

危险测试只能由授权实验室人员按硬件安全规程执行，平台文档不替代实验室 SOP。

## 9. 测试数据和隔离

- 使用合法校验位的测试 SN，并保留测试设备清单；不使用格式错误 SN 规避真实路径。
- 测试用户/电站/设备每例隔离，随机数据可复现（记录 seed）。
- 测试秘密只用 `TEST_*` 环境变量和明显测试值，不复用生产凭据。
- 时间测试固定 UTC/Asia-Shanghai、跨午夜、DST 地区（若产品支持）和边界秒。
- 性能测试数据与功能测试库分离；生产数据必须脱敏且经批准。

## 10. 覆盖率和质量阈值

覆盖率不是唯一门禁，但不得下降：

- 核心安全/协议/状态机分支：100% 需求场景覆盖。
- Go 变更包行覆盖建议≥80%，控制/权限/协议核心建议≥90%。
- React/Flutter 核心页面、store/service 和状态机有分支测试。
- 新缺陷必须先添加能复现的测试，再修复。
- flaky 视为缺陷；禁止无期限 retry 或忽略失败。

## 11. 缺陷等级

| 等级 | 示例 | 发布规则 |
|---|---|---|
| P0 | 过充放/断电安全、越权控制、真实秘密泄露、数据不可恢复 | 立即停止，禁止发布 |
| P1 | 核心功能错误、命令状态错误、告警丢失、迁移失败、OTA 无回滚 | 候选版本零遗留 |
| P2 | 有绕行的普通功能、性能/兼容性明显不达标 | 需负责人批准和期限 |
| P3 | 文案、轻微 UI、非关键维护性 | 可入计划 |

## 12. 发布退出条件

只有同时满足以下条件才能发布：

1. 所有 MUST 需求有实现和证据，追踪矩阵无空项。
2. 静态、单元、集成、契约、E2E、安全门禁通过，必需测试 0 Skip。
3. CS6K2 HIL 阻断用例通过，厂家安全包络签字。
4. P0/P1 为零；P2 有书面风险接受。
5. 性能 SLO、24h soak、容量和故障注入达到候选版本目标。
6. 凭据已轮换，secret scan/依赖/镜像无阻断高危。
7. 备份恢复、迁移和应用/OTA 回滚已演练。
8. 仪表盘、告警、runbook、值班、发布单和观察窗口就绪。

## 13. 可执行入口

查看将运行的门禁：

```powershell
powershell -ExecutionPolicy Bypass -File tools/cs6k2_audit.ps1 -Profile full -ListOnly
```

只执行仓库结构、CI 策略、凭据模式和迁移配对盘点（不调用 Go/npm/Flutter/Docker）：

```powershell
powershell -ExecutionPolicy Bypass -File tools/cs6k2_audit.ps1 -Profile static -InternalOnly
```

执行静态审查：

```powershell
powershell -ExecutionPolicy Bypass -File tools/cs6k2_audit.ps1 -Profile static
```

执行单元或集成：

```powershell
powershell -ExecutionPolicy Bypass -File tools/cs6k2_audit.ps1 -Profile unit
powershell -ExecutionPolicy Bypass -File tools/cs6k2_audit.ps1 -Profile integration
```

严格集成模式把任何 `--- SKIP:` 当失败；只有本地盘点可显式加 `-AllowSkippedIntegration`，该结果不能作为发布证据。
