## 2. 项目目标

### 2.1 业务目标

1. 支持厂家、系统运维/安装商和终端用户安全地监测、配置和维护 CS6K2。
2. 从设备遥测、告警、控制、OTA 到页面展示形成可追踪闭环。
3. 同一平台支持单机、单相并联和三相组网，型号差异通过能力模型表达。
4. 支撑从小规模试产到万级在线设备扩展，并为 10 万设备容量规划保留演进路径。
5. 所有影响供电、电池和固件的操作均可授权、可校验、可审计、可恢复。

### 2.2 非目标

- 系统端不替代 DSP/BMS 的本地硬件保护。
- V1 不提供任意 Modbus 寄存器读写、任意脚本执行或普通用户恢复出厂。
- 不把 5 秒全量遥测永久保留为热数据；历史查询按 3 分钟/小时/日/月分层。
- 不承诺未经硬件和认证确认的并网、防逆流、发电机干接点或负载切除功能。

## 3. 当前项目基线

| 层 | 当前实现 | 本项目要求 |
|---|---|---|
| Web | React 18、TypeScript、Ant Design、React Query、Zustand、Vitest | 系统支持端主工作台、能力驱动页面、风险交互 |
| App | Flutter、Dio、MQTT、BLoC/服务层 | 终端用户简化控制、实时数据、告警和配网 |
| API Gateway | Go、JWT、RBAC、限流、代理、Prometheus | 唯一公网 API 入口、对象级授权和安全头 |
| API Server | Go、Gin、PostgreSQL/TimescaleDB、Redis | 业务、控制编排、查询、告警、OTA、审计 |
| Device Server | Go、MQTT、Kafka/Redis、协议解析 | 设备消息接入、校验、标准化、命令响应 |
| Broker/流 | EMQX；Kafka Bridge 可选 | QoS、ACL、LWT、同设备有序、失败重试/DLQ |
| 数据 | PostgreSQL 18 + TimescaleDB 2.28、Redis 7 | 最新态、3分钟明细、聚合、事件、配置影子 |
| 部署 | Docker Compose、GitHub Actions、GHCR | 不可变镜像、迁移门禁、健康检查和回滚 |

## 4. 强制设计原则

| ID | 原则 | 验收方式 |
|---|---|---|
| SYS-001 | capability-first：型号/固件不支持的字段和命令不展示、不接受 | 型号能力 API 和负向测试 |
| SYS-002 | fail-closed：未知设备、型号、命令、角色或 schema 默认拒绝 | API/设备负向测试 |
| SYS-003 | device-final-authority：固件/BMS 对安全条件有最终否决权 | HIL 联调和拒绝码测试 |
| SYS-004 | desired≠reported：发送、ACK、执行、生效分别展示 | 状态机 E2E 测试 |
| SYS-005 | event-time-first：保留设备 `t`、服务端 `received_at` 和原始信封 | 数据库契约测试 |
| SYS-006 | idempotent：QoS 重投、HTTP 重试、乱序回复不得重复执行或倒退状态 | 重复/乱序测试 |
| SYS-007 | least-privilege：查看、普通控制、高风险配置、厂家服务权限分离 | RBAC 矩阵测试 |
| SYS-008 | immutable-release：镜像、协议、型号模板和电池模板版本化发布 | 发布证据审查 |
| SYS-009 | observable：关键链路有 metrics/log/trace/task_id/data_hash | 监控和追踪测试 |
| SYS-010 | reversible-change：迁移、配置和 OTA 有备份/回滚或补偿路径 | 演练报告 |

## 5. 角色与责任

| 角色 | 能力 | 明确禁止 |
|---|---|---|
| 厂家研发 | 型号、协议、安全包络、电池模板、固件签名 | 直接修改生产历史数据掩盖问题 |
| 厂家售后/超级管理员 | 跨租户诊断、高风险维护、OTA 灰度 | 无审计执行批量危险命令 |
| 平台管理员/运营商 | 用户、电站、设备、告警、报表、策略运营 | 修改厂家安全定值 |
| 安装商 | 所属设备安装配置、电池/AC 输入、并机调试 | 访问无归属设备、校准和恢复厂家密钥 |
| 终端用户 | 所有设备的日常模式、SOC 策略、时间计划 | 电池化学体系、并机相位、底层电压/电流保护 |
| 分享用户/访客 | 被授权数据的只读或显式受限能力 | 默认获得控制权 |
| 设备身份 | 仅发布自身上行主题、订阅自身下行主题 | 访问其他 SN 或管理 API |

RACI：产品负责人批准范围；硬件/固件负责人批准安全包络；架构负责人批准接口和数据；安全负责人批准威胁模型；测试负责人批准证据；发布经理作最终上线决策。

## 6. 版本与需求管理

- 需求使用稳定 ID：`FUN-*`、`UI-*`、`ARC-*`、`PER-*`、`SEC-*`、`DEP-*`、`E2E-*`。
- `MUST` 为 V1 发布阻断，`SHOULD` 为 V1.1，`MAY` 为后续规划。
- 每个 MUST 需求必须在追踪矩阵中有实现位置、自动测试或人工/HIL 证据。
- 需求变更必须说明设备协议、数据库、API、Web、App、运维和兼容性影响。
- 已发布 command code、field key、MQTT topic 和数据库迁移禁止复用旧语义。

## 8. 当前发布阻断项

1. 6.2kW 与 BMS 60A 充电/120A 放电能力尚未完成硬件核定。
2. 控制权限当前复用数据查看权限，且未配置型号/命令时存在兼容性放行。
3. `control-fields` 未向 App 提供真实 `parameter_schema`。
4. 仓库历史部署文档/脚本/测试中存在疑似真实凭据或硬编码口令，必须轮换并清理历史。
5. CI 的 Flutter analyze 非阻断，安全扫描非每次 PR 运行。
6. 当前集成测试在依赖服务不可达时 Skip，可产生假通过。
7. Makefile `lint-web` 调用的 npm `lint` 脚本当前不存在。
8. README、测试/开发/生产 Compose 对 PostgreSQL 版本、端口和拓扑描述不完全一致。

以上问题的检查方式和关闭证据见审查与测试计划。
