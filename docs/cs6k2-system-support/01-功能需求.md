# CS6K2 系统支持端功能需求

## 1. 产品范围和优先级

本文件描述系统功能，不定义底层电气安全值。控制参数范围以型号能力、电池模板和设备实时限制的交集为准。

优先级：MUST=V1 必须；SHOULD=V1.1；MAY=规划。

## 2. 账号、租户和权限

| ID | 级别 | 需求 | 验收标准 |
|---|---|---|---|
| FUN-AUTH-001 | MUST | 支持超级管理员、管理员/运营商、经销商、安装商、终端用户和访客/分享用户 | 前后端角色名称、role id 和权限一致 |
| FUN-AUTH-002 | MUST | 登录、刷新、退出、改密、找回密码形成完整会话生命周期 | refresh token 不可用 access token 替代；退出后令牌失效 |
| FUN-AUTH-003 | MUST | 菜单权限、API action 权限和设备/电站对象权限三层同时生效 | 直接请求隐藏接口仍返回 403 |
| FUN-AUTH-004 | MUST | 查看与控制分权；分享权限显式包含 read/control 范围和有效期 | 只读分享用户无法下发控制 |
| FUN-AUTH-005 | MUST | 高风险操作使用细粒度权限码和二次确认 | R3/R4 无权限、过期授权均拒绝 |
| FUN-AUTH-006 | SHOULD | 管理员 MFA、异常登录提醒和并发会话管理 | 可查看/撤销活跃会话 |

## 3. 设备接入和生命周期

| ID | 级别 | 需求 | 验收标准 |
|---|---|---|---|
| FUN-DEV-001 | MUST | 按 16 位 SN 规则接入，所有入口统一校验 | MQTT/info、绑定、导入、内部 API 均拒绝非法 SN |
| FUN-DEV-002 | MUST | 设备首次上线自动登记，但处于 unclaimed/pending 状态 | 未绑定设备不可被普通用户访问或控制 |
| FUN-DEV-003 | MUST | 支持绑定、解绑审批、转移、分配安装商、退役 | 生命周期变更有历史和操作者 |
| FUN-DEV-004 | MUST | 设备绑定型号、固件、协议版本和电池模板 | 未绑定型号默认拒绝控制 |
| FUN-DEV-005 | MUST | 在线状态综合 LWT、retained status、最后有效上行和服务健康 | 展示 online/offline/degraded/unknown 及原因 |
| FUN-DEV-006 | MUST | 配置 desired/reported/revision 和漂移状态 | 页面可查看差异、最后同步时间和关联 task_id |
| FUN-DEV-007 | SHOULD | 支持设备标签、批次、客户、区域和保修状态 | 可筛选和导出 |

## 4. 型号、协议和能力管理

| ID | 级别 | 需求 | 验收标准 |
|---|---|---|---|
| FUN-MODEL-001 | MUST | 型号基本信息、遥测字段、控制命令、协议版本分离管理 | 页面分为独立页签，职责不混用 |
| FUN-MODEL-002 | MUST | 协议 draft/released/retired 版本化；设备仅绑定 released | 已发布协议不可原地改数组下标/语义 |
| FUN-MODEL-003 | MUST | capability 决定字段、曲线、告警、命令和表单 | 不支持项 API 拒绝且 UI 不显示 |
| FUN-MODEL-004 | MUST | 型号激活前验证额定值、协议、字段、命令、schema、样例 | 验证报告无阻断项才可激活 |
| FUN-MODEL-005 | MUST | 电池模板版本化，包含化学体系、串数、BMS、温度和电流包络 | LiFePO4 模板不可使用铅酸均衡参数 |

## 5. 实时监测和历史数据

| ID | 级别 | 需求 | 验收标准 |
|---|---|---|---|
| FUN-TEL-001 | MUST | 展示 AC、电池、PV、系统、能量、并机、三相和电芯数据 | 数据单位、符号方向和质量状态一致 |
| FUN-TEL-002 | MUST | 实时页面展示数据年龄和质量；异常值不伪装为正常值 | OUT_OF_RANGE/CLOCK_INVALID 等有明确状态 |
| FUN-TEL-003 | MUST | 历史查询按 1h/6h/24h/7d/自定义自动选择粒度 | 返回 actual_granularity，不用明细撑长周期 |
| FUN-TEL-004 | MUST | 能量流满足功率方向和守恒校验，无法闭合时标记估算/异常 | 不展示物理不可能的正负方向 |
| FUN-TEL-005 | MUST | 离线补传写历史但不得覆盖较新的实时状态 | 乱序样本有质量标记 |
| FUN-TEL-006 | SHOULD | 支持设备/电站对比、日报/月报和导出任务 | 大导出异步、分页/流式、可取消 |

## 6. 告警、事件和工单

| ID | 级别 | 需求 | 验收标准 |
|---|---|---|---|
| FUN-ALM-001 | MUST | 告警使用 active/recovered 生命周期，设备事件和人工状态分离 | 恢复不能清除确认/处理记录 |
| FUN-ALM-002 | MUST | 告警去重、升级、恢复、确认、指派、处理、关闭全流程 | 重复 MQTT 事件不创建重复活动告警 |
| FUN-ALM-003 | MUST | 告警包含设备、来源、代码、级别、发生/恢复时间和遥测快照 | 快照缺失明确标记，不阻断事件入库 |
| FUN-ALM-004 | MUST | 支持通知策略、静默窗口、去重 TTL 和升级链 | 严重故障不可被永久静默 |
| FUN-ALM-005 | SHOULD | 告警可转工单，工单关联安装商、SLA、备件和处理记录 | 告警/工单双向跳转 |

## 7. 设备控制和能源策略

具体控制项见控制参数设计文档；本节定义业务行为。

| ID | 级别 | 需求 | 验收标准 |
|---|---|---|---|
| FUN-CTL-001 | MUST | 所有控制经 API Server 编排，App/Web 不直接向 MQTT 发布业务命令 | Broker ACL 拒绝 App 写 cmd topic |
| FUN-CTL-002 | MUST | 命令校验角色、型号、schema、状态、BMS、互锁、TTL 和 revision | 任一条件失败返回稳定拒绝码 |
| FUN-CTL-003 | MUST | 支持 pending/queued/sent/acknowledged/executing/success/failed/timeout/cancelled | 状态只能合法前进，迟到回复不倒退终态 |
| FUN-CTL-004 | MUST | `task_id` 端到端不变，设备对重复 task_id 幂等 | 重发不会重复切换继电器/写 Flash |
| FUN-CTL-005 | MUST | HTTP 接受不等于设备成功；reported 一致后才显示生效 | UI 分阶段显示 |
| FUN-CTL-006 | MUST | 强充/强放是带 TTL 的临时覆盖，互斥且受 SOC/BMS 限制 | 云端离线后设备本地仍按时退出 |
| FUN-CTL-007 | MUST | SOC、AC 充电和时间计划原子下发并带 revision | 并发修改返回 409/REVISION_CONFLICT |
| FUN-CTL-008 | MUST | 批量控制逐台授权、校验并创建 task_id | 一台失败不伪装为全批成功 |
| FUN-CTL-009 | MUST | 控制历史和配置事件保留操作前后值、来源和结果 | R2+ 审计可追溯 |

## 8. 并机和三相

| ID | 级别 | 需求 | 验收标准 |
|---|---|---|---|
| FUN-PAR-001 | MUST | 支持 standalone/single_phase/three_phase 能力 | 不支持并机的型号无相关入口 |
| FUN-PAR-002 | MUST | 组网使用安装向导和组级事务，不暴露多条裸命令 | 全组预检查、提交、同步、启动有统一结果 |
| FUN-PAR-003 | MUST | 唯一 master、machine_id 唯一、三相相位规则和成员归属校验 | 非本站设备、重复 ID/相位拒绝 |
| FUN-PAR-004 | MUST | 全组停机后才能修改拓扑；失败保持全组安全停机 | 不出现部分启动 |
| FUN-PAR-005 | MUST | 三相页面展示相电压/电流/功率、线电压、频率和不平衡度 | 数据来源为 master，成员状态可下钻 |

## 9. OTA 和维护

| ID | 级别 | 需求 | 验收标准 |
|---|---|---|---|
| FUN-OTA-001 | MUST | 固件包包含 target、型号、版本、大小、哈希、签名、兼容范围 | 未签名或型号不匹配拒绝发布/安装 |
| FUN-OTA-002 | MUST | OTA 前检查在线、供电、SOC、故障、并机状态和空间 | 前置条件有明确结果 |
| FUN-OTA-003 | MUST | 支持灰度批次、并发限制、暂停、取消、失败阈值自动停止 | 失败率越界不继续扩大 |
| FUN-OTA-004 | MUST | 进度状态不可倒退，升级后验证版本和健康；失败可回滚 | 任务最终态和设备实际版本一致 |
| FUN-OTA-005 | SHOULD | 支持维护窗口、用户通知和安装商确认 | 离网关键负载场景避免无计划重启 |
| FUN-MNT-001 | MUST | 自检、诊断快照、故障复位、受控重启分开授权 | `restart` 不再承担多个语义 |

## 10. 电站、统计和运营

| ID | 级别 | 需求 | 验收标准 |
|---|---|---|---|
| FUN-STA-001 | MUST | 电站管理设备、位置、时区、容量、安装商和所有者 | 时间计划使用电站时区 |
| FUN-STA-002 | MUST | 电站聚合发电、充电、放电、负载和可用率 | 聚合不重复计并机成员/主机数据 |
| FUN-STA-003 | SHOULD | 日/月/年报、异常设备排行和运营看板 | 指标定义可追踪到原始字段 |
| FUN-STA-004 | SHOULD | 策略模板可批量分配同型号/同固件/同电池模板设备 | 差异设备被排除并报告原因 |

## 11. 审计、通知和支持

| ID | 级别 | 需求 | 验收标准 |
|---|---|---|---|
| FUN-AUD-001 | MUST | 登录、权限、设备生命周期、配置、控制、OTA、导出均写审计 | 失败和拒绝也记录，不含秘密值 |
| FUN-AUD-002 | MUST | 审计记录不可由普通业务接口修改/删除 | 管理员也只能查询 |
| FUN-NOT-001 | MUST | 站内、推送、邮件按用户偏好和告警策略发送 | 去重且记录发送结果 |
| FUN-SUP-001 | SHOULD | 售后可生成脱敏诊断包和关联工单 | 不包含 token、密码、WiFi 密码或私钥 |

## 12. 功能完成定义

每项功能只有同时满足以下条件才算完成：

1. 需求 ID、API/协议 schema、数据表和 UI 状态已评审。
2. 正常、边界、权限、异常、重复、乱序、离线和恢复路径均实现。
3. 自动测试与必要的硬件在环测试已通过。
4. metrics、日志、trace、审计和运维说明齐全。
5. 向前兼容和回滚/补偿路径已验证。
