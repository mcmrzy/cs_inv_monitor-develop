# CS6K2 系统支持端安全要求

## 1. 安全目标

1. 未授权主体不能读取或控制不属于自己的设备。
2. 云端、App、网络攻击不能绕过逆变器/BMS 本地保护。
3. MQTT、HTTP、内部服务和 OTA 的身份、完整性和重放防护闭环。
4. 凭据、个人数据、设备数据和审计记录按最小权限保护。
5. 发生攻击或凭据泄露时可发现、吊销、隔离、恢复和追溯。

## 2. 威胁模型

重点威胁：

- 伪造用户 JWT、越权访问 SN、只读分享获得控制。
- 伪造设备、跨 SN 发布、重放旧命令/回复、篡改 MQTT 数据。
- 恶意参数造成断电、电池过充放、并机冲突或永久强充/强放。
- OTA 包被替换、降级攻击、错误型号刷写或升级中断。
- 仓库/日志/文档泄露密码、Token、服务器账号、WiFi 或客户信息。
- API 注入、XSS、CSRF、SSRF、文件上传、DoS、供应链漏洞。
- 管理员误操作、批量任务扩大影响、审计被删除或篡改。

## 3. 身份、会话和授权

| ID | 级别 | 要求 |
|---|---|---|
| SEC-IAM-001 | MUST | 生产 JWT secret 至少 256 bit 随机，来自密钥托管，不在镜像/仓库/日志 |
| SEC-IAM-002 | MUST | Access/refresh token 类型、aud/iss/exp/jti 明确；refresh rotation 和重用检测 |
| SEC-IAM-003 | MUST | 退出、改密、角色变更、账号禁用可吊销会话；缓存及时失效 |
| SEC-IAM-004 | MUST | Gateway 和 API 都执行授权；对象级检查以 user/tenant/station/device 关系为准 |
| SEC-IAM-005 | MUST | 只有 super admin 可无条件系统级旁路；其他角色都按 permission 和对象范围 |
| SEC-IAM-006 | MUST | 查看、基础控制、断电控制、安装配置、并机、厂家服务、OTA 分权 |
| SEC-IAM-007 | SHOULD | 管理员/厂家售后启用 MFA；高风险服务授权短时有效、一次性使用 |
| SEC-IAM-008 | MUST | 登录/验证码/刷新/导出/控制同时按账号、IP、租户或设备维度限流 |

Web 优先使用 Secure、HttpOnly、SameSite Cookie 或经评审的内存令牌模式；禁止长期把 refresh token 明文持久化到可被任意 JS 读取的 localStorage。

## 4. 设备和 MQTT 安全

| ID | 级别 | 要求 |
|---|---|---|
| SEC-IOT-001 | MUST | 每台设备独立身份/密钥或客户端证书，禁止全量设备共享长期万能凭据 |
| SEC-IOT-002 | MUST | Broker ACL：设备仅 publish `cs_inv/{own_sn}/...` 上行、subscribe 自身 cmd/ota |
| SEC-IOT-003 | MUST | App 默认不直接订阅无授权 SN；短期 token 只包含允许的 topic 和过期时间 |
| SEC-IOT-004 | MUST | MQTT 使用 TLS，校验证书链或经管理的证书 pin；生产禁止 skip verify |
| SEC-IOT-005 | MUST | 命令包含 task_id、创建时间、过期时间、协议版本；设备拒绝过期/重复/未知命令 |
| SEC-IOT-006 | MUST | 设备密钥支持生产注入、轮换、吊销和丢失设备隔离，不通过普通 OTA 明文导出 |
| SEC-IOT-007 | SHOULD | 高安全版本使用安全启动、固件签名验证和受保护密钥存储 |

## 5. 控制安全

| ID | 要求 |
|---|---|
| SEC-CTL-001 | 所有命令在型号能力中登记；无型号、空命令表、未知命令默认拒绝 |
| SEC-CTL-002 | 删除 `systemCommands` 式无条件旁路；恢复出厂/OTA/重启分别授权 |
| SEC-CTL-003 | 参数同时做 schema、关系、当前状态、BMS、温度、SOC、并机和 revision 校验 |
| SEC-CTL-004 | 固件拥有最终否决权，云端允许范围只能比本地保护更严格 |
| SEC-CTL-005 | R2+ 二次确认和审计；R3 要求停机/服务授权；R4 不暴露普通 API |
| SEC-CTL-006 | 强充/强放、继电器手动覆盖必须 TTL；设备离线时本地到期 |
| SEC-CTL-007 | 批量控制设置最大设备数、并发、速率、失败阈值和紧急停止 |
| SEC-CTL-008 | 高风险命令不盲目自动重试；超时先查询状态 |

## 6. API 和 Web 安全

- 只允许配置白名单 Origin；带凭据时禁止 `Access-Control-Allow-Origin: *`。
- 生产全站 HTTPS，HSTS、CSP、frame-ancestors、nosniff、Referrer-Policy 和 Permissions-Policy。
- 所有输入有长度、类型、枚举、格式和业务验证；数据库参数化查询。
- 文件上传限制类型、大小、扩展名、实际 MIME、病毒扫描和对象存储隔离。
- 外部 URL（天气、OTA、Webhook）防 SSRF：域名/协议白名单、禁止私网和重定向绕过。
- 错误不回传堆栈、SQL、内部地址和凭据；日志对 token/手机号/邮箱/IP 按策略脱敏。
- 管理后台依赖 CSP；富文本、错误消息和设备字符串按不可信输入处理。

## 7. 内部服务和数据安全

| ID | 要求 |
|---|---|
| SEC-SVC-001 | 内部 API 使用独立强密钥或 mTLS；密钥为空时 fail-closed |
| SEC-SVC-002 | DB/Redis/Kafka/Device Server 不直接暴露公网；网络分段和防火墙最小开放 |
| SEC-SVC-003 | 生产数据库连接至少 verify-ca/verify-full（跨主机）；备份加密 |
| SEC-SVC-004 | PostgreSQL 角色分离：迁移、应用读写、只读报表、备份各自账号 |
| SEC-SVC-005 | Redis 启用认证/TLS（跨主机），不存永久唯一事实；高敏 token 值加密/哈希 |
| SEC-DAT-001 | 数据按租户和设备对象授权；导出也执行相同授权并有有效期 |
| SEC-DAT-002 | 审计追加写、限制删除；关键审计外送到独立存储或日志平台 |
| SEC-DAT-003 | 定义个人数据、设备数据、位置数据、诊断数据的保留和删除规则 |

## 8. 凭据和仓库治理

当前仓库的历史部署文档、辅助 Python、测试或报告中存在疑似真实服务器信息和硬编码凭据。这不是通过 `.gitignore` 可以修复的问题。

发布前必须：

1. 盘点并立即轮换所有可能出现过的 SSH、数据库、Redis、JWT、MQTT、SMTP、Webhook、Jenkins 凭据。
2. 将生产秘密迁入服务器侧 secret 文件（权限 600）、Docker secret、Vault/KMS 或 CI secret。
3. 从当前工作树删除硬编码，测试改用 `TEST_*` 环境变量和明显测试值。
4. 使用 secret scanner 检查完整 Git 历史；按安全负责人决定历史重写和克隆端处置。
5. 日志/报告只记录 secret 名称、版本和指纹，不打印值。
6. 建立 90 天或风险驱动的轮换、紧急吊销和所有者清单。

| ID | 要求 |
|---|---|
| SEC-SEC-001 | PR 和 main 每次运行 secret scan；发现高置信秘密阻断 |
| SEC-SEC-002 | `.env.example` 只放 CHANGE_ME；真实 `.env` 不进入版本控制 |
| SEC-SEC-003 | 代码和运维脚本不得打印密码、hash、token、私钥或完整连接串 |
| SEC-SEC-004 | 测试凭据与生产完全隔离，不能恰好复用生产值 |

## 9. OTA 和供应链

| ID | 要求 |
|---|---|
| SEC-OTA-001 | 固件采用厂家离线/受控私钥签名，设备内置公钥验证 |
| SEC-OTA-002 | manifest 绑定 target、型号、硬件版本、版本、hash、size 和最低允许版本 |
| SEC-OTA-003 | 防降级；紧急回滚使用显式授权的已签名版本 |
| SEC-OTA-004 | 下载使用 TLS；即使 TLS 成功也必须验证内容签名和 hash |
| SEC-SUP-001 | CI 固定 action/镜像/依赖版本，生成 SBOM，扫描依赖和镜像漏洞 |
| SEC-SUP-002 | 发布镜像使用 SHA/digest，不由生产直接部署可变 `latest` |
| SEC-SUP-003 | 构建与部署账号分离，最小 GitHub permissions，保护 main 和生产环境审批 |

## 10. 安全监控和事件响应

必须告警：登录暴破、refresh 重用、权限拒绝突增、跨 SN MQTT、未知命令、R3/R4 操作、批量控制、OTA 失败/回滚、secret scan、审计中断、异常导出、TLS 失败。

事件流程：发现 → 分级 → 隔离账号/设备/凭据 → 保全日志 → 修复/轮换 → 恢复 → 复盘。设备控制安全事件必须有“停止批量任务、禁止远控、设备本地自治”的开关。

## 11. 安全测试和发布门禁

| ID | 测试 |
|---|---|
| T-SEC-001 | 全角色×资源×action×对象归属矩阵，包含直接 API 越权 |
| T-SEC-002 | JWT 篡改、过期、类型混淆、refresh 重用、吊销和缓存失效 |
| T-SEC-003 | MQTT 跨 SN publish/subscribe、过期 token、TLS 错误证书、重放 |
| T-SEC-004 | 未知命令、空型号、空 capability、越界/关系/BMS/模式联锁 |
| T-SEC-005 | OWASP API/Web 常见注入、XSS、CSRF、SSRF、上传、限流和大包 |
| T-SEC-006 | OTA 签名错误、hash 错、错误型号、降级、中断和回滚 |
| T-SEC-007 | secret、SAST、依赖、容器、IaC 扫描 |
| T-SEC-008 | 第三方渗透测试和高危关闭复测（量产/重大版本） |

存在未轮换真实秘密、控制 fail-open、生产 TLS 验证关闭、越权控制、高危漏洞或 OTA 未验签时，一律禁止发布。
