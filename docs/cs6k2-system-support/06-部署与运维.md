# CS6K2 系统支持端部署与运维要求

## 1. 环境和配置

| 环境 | 目的 | 数据/外部服务 | 发布方式 |
|---|---|---|---|
| local | 开发和单元测试 | 本地匿名/合成数据 | 本地进程或开发 Compose |
| test | 自动集成 | 独立 PG/Redis/EMQX/Kafka（按链路） | CI 临时环境 |
| staging | 全链路、HIL、性能和发布演练 | 脱敏/合成，独立设备和 Broker | 与生产同镜像/拓扑 |
| production | 正式服务 | 正式数据和设备 | 审批后的不可变镜像 |

| ID | 要求 |
|---|---|
| DEP-ENV-001 | 每个环境独立数据库、Redis、Broker namespace、凭据和域名 |
| DEP-ENV-002 | 配置模板无秘密；生产秘密只在部署时注入，容器内只读 |
| DEP-ENV-003 | 启动时校验必需配置、占位值、TLS 模式和服务 URL，错误即退出 |
| DEP-ENV-004 | 统一版本基线：Go 1.25、Node 24、Flutter 3.44、PG18/Timescale 2.28（变更需 ADR） |
| DEP-ENV-005 | README、Compose、CI 和生产清单使用同一版本来源，禁止文档漂移 |

## 2. 生产拓扑

公网仅开放 Nginx/负载均衡器的 443 和 MQTT Broker 的设备入口。Gateway、API Server、Device Server、PostgreSQL、Redis、Kafka、Prometheus 不直接暴露公网。

```text
Internet/App/Web ─ HTTPS 443 ─ LB/Nginx ─ API Gateway ─ API Server
CS6K2 ─ MQTT TLS 8883 ─ EMQX ─ [Kafka/Bridge] ─ Device Server
                                     │                │
                                     └──── internal network ─ PG/Redis
Metrics/Logs/Traces ─ separate observability network / authenticated access
```

固件下载可由受控对象存储/CDN或 API 提供，但必须使用短期签名 URL、TLS、限速、签名校验；不能以“避免网关卡顿”为由永久暴露业务 API Server 端口。

## 3. 镜像和制品

| ID | 要求 |
|---|---|
| DEP-ART-001 | 每个服务产生独立镜像，标签含 Git SHA，部署记录 digest |
| DEP-ART-002 | 禁止生产依赖可变 `latest`；latest 只能作便捷标签，不作证据 |
| DEP-ART-003 | 多阶段构建、非 root、只读根文件系统（必要目录挂卷）、最小基础镜像 |
| DEP-ART-004 | 生成 SBOM、漏洞扫描、许可证报告和构建 provenance |
| DEP-ART-005 | 前端静态文件带内容 hash；API/协议兼容性在发布前校验 |
| DEP-ART-006 | 固件、App、服务镜像分别签名和发布，制品不可覆盖 |

## 4. CI 门禁

每个 PR 必须：

1. Go build/vet/test-race；四个 module 都执行。
2. React `npm ci`、`tsc`、build、Vitest/coverage。
3. Flutter analyze（不得 `|| true`）、test/coverage、至少 Android 构建冒烟。
4. 数据库空库 up、已有库 upgrade、down/恢复策略和 schema 合同测试。
5. secret、SAST、dependency、container、IaC scan。
6. OpenAPI/协议/schema/迁移配对和需求追踪检查。

main 合并后才构建一次镜像；staging 和 production 提升同一 digest，禁止生产重新编译。

## 5. 数据库迁移

| ID | 要求 |
|---|---|
| DEP-DB-001 | 迁移编号唯一、up/down 配对；不可修改已在生产执行的迁移 |
| DEP-DB-002 | 空库从基线到最新、旧生产快照到最新均自动验证 |
| DEP-DB-003 | API 启动完成迁移前 readiness=false；失败不得报告健康 |
| DEP-DB-004 | 大表 DDL 评估锁、WAL、磁盘和超时；必要时 expand/migrate/contract |
| DEP-DB-005 | 删除表/列先停止读写并观察至少一个兼容窗口，保留回滚证据 |
| DEP-DB-006 | PG major 升级独立演练，不能让新容器直接复用不兼容数据目录 |

迁移步骤：预检查 → 备份/快照 → 只读验证 → 执行 expand → 部署兼容代码 → 回填/校验 → 切换 → contract（后续版本）→ 记录版本。

## 6. 发布流程

```mermaid
flowchart LR
  A["需求/变更冻结"] --> B["CI 全门禁"]
  B --> C["构建并签名镜像"]
  C --> D["Staging 部署"]
  D --> E["自动 E2E + HIL + 安全冒烟"]
  E --> F["备份/回滚点/发布审批"]
  F --> G["生产 canary/单实例"]
  G --> H["指标与业务验证"]
  H --> I["分批滚动"]
  I --> J["发布后观察与关闭"]
```

| ID | 要求 |
|---|---|
| DEP-REL-001 | 发布单包含 SHA/digest、迁移版本、配置版本、负责人、风险和回滚步骤 |
| DEP-REL-002 | 发布前冻结批量控制/OTA或明确协调，避免并发高风险操作 |
| DEP-REL-003 | Gateway/API 可滚动；Device Server 依赖共享订阅/分区再均衡验证无重复副作用 |
| DEP-REL-004 | canary 观察 API、消息、DB、命令、告警、业务指标，不只看容器 running |
| DEP-REL-005 | 超过错误/延迟/积压/控制失败阈值自动停止扩大 |

## 7. 健康检查

区分：

- liveness：进程事件循环可响应，不因外部 Broker 暂时离线反复重启。
- readiness：接受新业务是否安全；迁移未完成、关键 DB 不可用时 false。
- dependency health：PG、Redis、MQTT、Kafka 的状态、更新时间和降级原因。

| 服务 | liveness | readiness |
|---|---|---|
| Gateway | HTTP 进程 | Redis/路由配置可用；后端状态单独展示 |
| API Server | HTTP 进程 | schema 最新、DB 可写；Redis 可降级时注明 |
| Device Server | HTTP/消费者循环 | 选定主通道可消费、DB 可持久化；MQTT 状态独立 |
| Bridge | HTTP 进程 | Kafka 可写、配置有效 |

当前 K8s/Compose 的 `/health` 必须核对是否符合上述语义，避免把 liveness 和 readiness 混为一条。

## 8. 备份、恢复和灾备

| ID | 目标 |
|---|---|
| DEP-DR-001 | 关系/事件/配置 RPO≤15分钟，RTO≤4小时（V1 目标，业务可提高） |
| DEP-DR-002 | 遥测明细 RPO≤1小时；小时/日聚合可从明细或备份重建 |
| DEP-DR-003 | 每日全量/增量或连续归档；备份加密、跨故障域、有限期不可变 |
| DEP-DR-004 | 每季度恢复演练到隔离环境，验证行数、关键约束、登录、查询和命令审计 |
| DEP-DR-005 | Redis 不作为备份对象的唯一事实；可从 PG/设备重新构建 |

恢复顺序：基础设施 → PG/Timescale → schema 校验 → Redis 清空/重建 → API → Device Server 消费位点 → Gateway/Web → 业务抽样。恢复期间禁止危险控制，直到设备 reported state 重新同步。

## 9. 回滚

- 应用回滚使用上一已验证 digest。
- 数据库优先前向修复；仅在 down 安全、已演练且无新数据语义冲突时执行 down。
- 协议/型号/电池模板通过版本回退绑定，不原地覆盖。
- OTA 使用设备 A/B 或 bootloader 回滚；并机组按组级策略。
- 回滚后运行同一冒烟、数据一致性和安全检查。

## 10. 可观测性和告警

必须有仪表盘：

1. 系统总览：请求、错误、延迟、实例、重启、资源。
2. 设备接入：在线数、消息率、重复/乱序/无效、接入延迟、Broker。
3. 数据库：连接、慢查询、锁、WAL、磁盘、chunk/压缩/聚合。
4. 控制：stage、ACK/完成时延、拒绝码、超时和漂移。
5. 告警/通知：事件率、活动数、通知失败和积压。
6. OTA：版本分布、进度、失败和回滚。

告警有 owner、severity、runbook、抑制规则和升级路径；监控系统自身不可用也要被外部探测发现。

## 11. 运维 Runbook

至少建立：

- DB 连接耗尽/磁盘高水位/锁等待。
- Redis 故障和缓存重建。
- MQTT Broker 断连、证书过期、跨 SN ACL 告警。
- Kafka lag/DLQ/分区不平衡。
- 遥测停止、告警风暴、命令超时突增。
- OTA 自动暂停/回滚。
- 凭据泄露和紧急轮换。
- 设备控制安全事件：立即禁止远控/批量任务。

## 12. 当前部署整改重点

1. 清理并轮换历史文档、部署脚本和测试中的硬编码凭据。
2. 统一 PG16/PG18 文档和数据目录，不再出现不一致基线。
3. 生产只暴露 443 和必要 MQTT TLS 入口；移除 API Server 固件下载直出端口或加受控边界。
4. CD 部署固定 digest、运行完整迁移集合检查，不只检查少数编号。
5. Flutter analyze、安全扫描和集成服务 Skip 改为阻断。
6. 生产 Compose 补齐独立 readiness、备份任务、Prometheus/日志/trace 和证书挂载。
