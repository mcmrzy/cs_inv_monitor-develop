# INV-MQTT 架构升级任务清单

> 创建日期: 2026-05-22  
> 更新日期: 2026-05-22  
> 当前架构: 设备(ESP32) → EMQX → inv_device_server(Go) → PostgreSQL + Redis → inv_api_server → App  
> 升级目标: 万级设备接入，消除单点，安全闭环

---

## 最终架构（升级后）

```
                            ┌── JWT 认证（HS256）──┐
                            │   Secret 一致          │
                            ▼                       ▼
App ── MQTT (password=JWT) ──→ EMQX ←── 设备(ESP32) 数据上报
  │                              │
  │                       内置 JWT 验签
  │                       过期自动断连
  │                              │
  └── HTTP REST ────────→ API Server ──── 历史/统计数据 + 命令下发
                              │
                              ▼
                         PostgreSQL + Redis
```

**原则**：实时走 MQTT 直连（低延迟），历史走 HTTP API（按需查询），安全靠 EMQX 内置 JWT 认证。

## Phase 1：EMQX 内置 JWT 认证 + App 鉴权改造（Week 1 · 🔴 最高优先级）

> **目标**：App MQTT 连接从全局共享密码切换到 JWT token 认证，EMQX 内置 JWT 验签，零代码写回调

### 1.1 EMQX Dashboard 配置 JWT 认证

| 字段 | 值 | 说明 |
|------|-----|------|
| 认证方式 | `JWT` | |
| JWT 来自于 | `password` | App 连接时 password 字段传 JWT token |
| 加密方式 | `hmac-based` | 对应 API Server 的 HS256 签名 |
| Secret | `CSKJ_INV_APP_SERVER_APP_MQTT_KEY` | 与 [config.yaml](file:///d:/INV-MQTT/inv_api_server/config.yaml#L25) 的 `jwt.secret` 一致 |
| Secret 使用 Base64 编码 | **不勾选** | Go 代码用明文 bytes 签名，EMQX 也用明文验签 |
| 过期后断开连接 | ✅ 勾选 | token 到期自动踢掉客户端 |
| Payload | 无需额外配置 | 标准 `exp`/`iat`/`iss` 声明即可 |

## 技术选型定案

| 选择 | 结论 | 理由 |
|------|------|------|
| MQTT Broker | **EMQX**（已有） | 单节点百万连接，内置 JWT 认证 |
| App 鉴权 | **EMQX 内置 JWT（password 模式，HS256）** | 零回调代码，API Server 和 EMQX 共用 Secret |
| 设备端鉴权（远期） | **EMQX JWT JWKS 模式（RS256）** | 私钥在服务端，设备丢了公钥无法伪造 token |
| App 实时推送 | **MQTT 直连 EMQX + JWT 认证** | 最低延迟，EMQX 内置 JWT 已解决安全 |
| App 历史数据 | **HTTP REST → API Server** | 保持不变 |
| 消息缓冲 | **Redis Streams** | 已有 Redis 基础设施，消费组+ACK 满足需求 |
| 时序数据库 | **TimescaleDB** | PG 插件，一行 SQL 切超表，迁移成本最低 |
| 协议 QoS | **不碰 QoS 2** | 协议层兜底不如消费侧兜底 |
| 字段映射 | **EMQX Rule Engine** | 减少 Go 代码，规则热更新 |
