1. 范围与定案

本文定义辰烁科技 48V 离网逆变器（型号 CS-I10-6k2）从设备到云端的 MQTT V1 协议，以及服务端的解析、缓存、时序存储、事件存储和质量控制规则。本文已经合并设备参数、产品量程、故障码、控制互锁、并机和 OTA 规范，正式取代《系统参数规范_48V离网逆变器》；设备固件、ESP32 固件、Device Server、API、Web/App 与测试程序只以本文作为交付基线。

单一实时快照整机实时数据统一由 heartbeat 上报，每 180 秒一次。
单一上行信封所有设备上行消息统一使用外层 t / v / data 结构。
单一字段版本Heartbeat 固定为 ac[8]、bat[23]、pv[7]、sys[11]、eng[12]、cells[2]。
规范用语：“必须”表示强制要求；“应”表示无特殊理由不得偏离；“可”表示可选能力。示例中的数值仅用于说明格式，不构成设备量程定义。

3. MQTT 通用规范

3.1 Topic 命名

设备根 Topic 固定为 cs_inv/{sn}。{sn} 为设备唯一序列号，只允许字母、数字、短横线和下划线，最大 50 字符。服务端以 Topic 中的 SN 为准，Payload 不重复传输 SN。

3.2 上行信封

{
  "t": 1783676930,
  "v": 1,
  "data": {}
}
字段类型必填定义
tint64是业务采样或事件生成时间，Unix 秒，UTC。除 LWT 外必须大于 0。
vuint16是协议主版本，本文固定为 1；属于信封元数据。
dataobject是当前 Topic 的业务对象。时间和协议版本不得放入此对象。
  • 编码必须为 UTF-8 JSON；禁止 NaN、Infinity 和字符串形式的数值。
  • t、v 是信封元数据；data 只保存业务字段。所有设备上行 Topic 均使用相同字段位置。
  • 固定数组的长度、下标、数据类型和单位永久冻结。暂不可用的值必须填 null，不得删除元素；数值 0 是有效业务值。
  • 实时遥测使用工程单位；控制命令参数使用整数缩放值，缩放规则由命令定义。
  • 上报失败后不得补发已经过期的实时快照。告警、命令结果和配置状态必须保留到确认投递。
  • 建议单条 Payload 不超过 16 KiB；超过限制时设备必须拒绝发送并记录本地错误。

3.3 上行去重与排序

V1 不使用全局消息序号。服务端按消息自身的业务标识去重；对于没有业务修订号的周期遥测，只在短时间窗口内消除内容完全相同的 QoS 1 重复投递，不建立跨设备重启的永久唯一约束。

消息去重/排序依据规则
heartbeatsn + t + data_hash相同采样时间且业务内容相同视为重复；按 t 排序。
infoSN按 SN UPSERT，信息变化保留审计记录。
alarmsn + source + code + state + t相同告警状态和事件时间视为重复。
configsn + rev只接受不小于当前 reported_rev 的配置。
parallel主机 SN + t + data_hash完整拓扑快照按采样时间排序，相同内容短期去重。
three_phase主机 SN + t + data_hash按采样时间排序；相同内容短期去重。
cmd/responsetask_id + stage同一任务阶段幂等，completed 终态不可回退。
ota/statustask_id + state + progress相同状态和进度幂等，终态不可回退。
statusBroker 接收时间使用 Retain、LWT 和最近有效上行时间判断在线。

data_hash 由服务端对规范化 JSON 的 data 对象计算，不由设备发送。建议重复检测窗口为 10 分钟；窗口外的相同数据仍可作为独立历史样本保存。

3.4 下行命令信封

{
  "v": 1,
  "task_id": "019f5b9e-7bd7-7e50-a14d-5c8a74235a10",
  "t": 1783676930,
  "cmd": "set_power_limit",
  "args": [5000]
}

task_id 必须全局唯一；设备必须以该字段实现幂等。命令创建后超过 300 秒尚未开始执行时，设备应返回 EXPIRED，不得执行。

8.2 状态上报

{
  "t": 1783676990,
  "v": 1,
  "data": {
    "task_id": "ota_019f5b",

state 依次为 accepted、downloading、verifying、installing、rebooting、succeeded；任一可取消阶段可转为 cancelled，任一阶段可转为 failed 或 rolled_back。进度范围为 0～100，变化达到 5% 或阶段变化时上报。升级失败必须保留明确错误码和原因；启动失败或新固件健康检查失败时必须回滚。

data 字段类型必填定义与约束
task_idstring是对应 OTA 命令的任务 ID。
targetstring是升级目标。
statestring是当前升级阶段或终态。
progressuint8是0～100；阶段变化时即使百分比未变也必须上报。
versionstring是目标版本。
codestring是OK 或 8.4 节错误码。
messagestring是可读状态或失败原因；成功时可为空字符串。
取消规则：accepted、downloading 和尚未擦写目标分区的 verifying 阶段允许取消，设备必须停止任务、清理临时文件并上报 cancelled。进入 installing、rebooting 或任何终态后禁止取消，回复 NOT_CANCELLABLE。重复 cancel 必须幂等返回当前终态。

8.3 升级路径与大小限制

目标升级路径最大固件执行要求
espESP32 下载 → Flash → 重启1 MiB写入非活动分区，启动健康检查失败时回滚。
armESP32 下载 → UART → ARM IAP256 KiBARM 校验完整镜像后切换。
dspESP32 下载 → UART → ARM → DSP256 KiBARM 负责分片、中转、校验和重试。
bmsESP32 下载 → UART → ARM → RS485 → BMS128 KiB升级期间必须保持电池安全状态和可靠供电。

设备必须先校验 HTTPS 证书、文件大小、SHA-256、Ed25519 发布签名、目标芯片和 security_version，全部通过后才允许写入。除经物理维护授权的恢复流程外，目标安全版本低于设备已确认安全版本时必须返回 ROLLBACK_BLOCKED。新固件启动后必须在 60 秒内完成自检、恢复通信并通过 info 上报目标版本；否则判定 HEALTH_CHECK_FAILED 并回滚。

9.4 Redis 与在线判定

  • telemetry:latest:{sn}：最近一条解析后的 Heartbeat，TTL 420 秒。
  • device:heartbeat:{sn}：任一有效设备上行消息均刷新，TTL 420 秒。
  • command:{task_id}：命令当前阶段，TTL 24 小时；数据库日志为最终事实来源。
  • 在线判定优先使用有效 status 与 heartbeat key；超过 420 秒没有有效上行数据即判定离线。业务健康状态必须另外依据 work_state、故障和数据新鲜度判断。

10. 数据质量、安全与可靠性

10.1 质量标记

位名称触发条件处理
0x0001NULL_VALUE固定字段为 null保留记录，不参与该字段聚合
0x0002OUT_OF_RANGE超出型号量程保留原值，禁止作为正常值展示
0x0004CLOCK_INVALID设备时间为 0、偏差过大或倒退以 received_at 入库并保留原 t
0x0008OUT_OF_ORDER采样时间早于当前最新样本可入历史库，不覆盖最新缓存
0x0010COUNTER_RESET累计量倒退或复位切断增量计算
0x0020COMM_FAULTARM、BMS 或传感器通信异常相关字段不得伪造为 0

JSON 非法、版本不支持、数组长度错误、类型错误等结构性问题不能写入正常遥测表，必须写入 device_ingest_errors 并计入监控指标。QoS 1 重复投递按 3.3 节的 Topic 业务键处理，但重复消息仍要刷新连接活跃时间；采样时间较旧的迟到消息不得覆盖最新缓存。

10.2 安全要求

  • 生产环境必须使用 MQTT over TLS；设备凭据必须按设备唯一，不得使用全局共享密码。
  • Broker ACL 必须限制设备只能发布自身 cs_inv/{sn}/...，且只能订阅自身 cmd 与 ota/cmd。
  • OTA URL 必须为 HTTPS 且严格校验证书；固件必须校验 SHA-256、Ed25519 发布签名和单调安全版本，禁止未授权降级。
  • 服务端日志不得记录设备密钥、访问令牌或完整授权头；控制、配置和 OTA 必须留审计日志。
  • 设备断网不影响本地保护与既有运行策略；重连采用指数退避并加入随机抖动。

11. 版本与兼容规则

  • V1 的数组下标、长度、类型和单位永久冻结，不得在尾部追加字段。
  • 新增可选信息优先使用新的具名对象或新 Topic；破坏性变化必须发布 V2。
  • 设备必须忽略未知的 JSON 顶层键；服务端按设备型号与外层 v 选择映射。
  • 旧 Topic 或旧信封只允许在服务端兼容层接收，不得由采用本文版本的新设备继续发送。
  • 协议升级必须同步更新设备固件、ESP32、Device Server、API、Web/App、字段模型和自动化测试样例。
禁止的 V1 变体：Heartbeat 中包含 brt；使用 bat[21]、sys[10] 或 eng[8]；使用独立 bms_diag；使用 {"data":...,"timestamp":...}；把 v 放入 data；把业务字段放在 data 外层。

12. 交付验收标准

  1. 所有上行 Topic 使用外层 {t,v,data} 信封，data 内不重复时间和协议版本。
  2. Heartbeat 精确包含 ac[8]、bat[23]、pv[7]、sys[11]、eng[12]、cells[2]，不包含 brt。
  3. 设备启动和 query_telemetry 均能触发 Heartbeat；稳定运行每 180 秒上报，允许调度误差 ±5 秒。
  4. 告警发生与恢复形成闭环；网络状态使用 Retain 和 LWT；业务健康与网络在线分开判断。
  5. 命令在 2 秒内 ACK，按 task_id 幂等，有最终结果；参数范围、安全互锁、超时和过期机制有效。
  6. 配置只有在设备上报新 rev 后更新 reported；desired 与 reported 可追溯。
  7. OTA 完成下载校验、目标校验、进度上报、失败原因、重启健康检查和回滚测试。
  8. 服务端完成数组到标准字段映射；原始信封、采样时间、接收时间、data_hash 和质量标记可查询。
  9. QoS 1 重复、迟到、时间异常、数组长度错误、字段越界、累计量复位和通信异常均有自动化测试。
  10. TimescaleDB 压缩与聚合、Redis TTL、告警闭环、命令日志和接入错误表均通过集成测试。
交付结论：设备端、服务端和应用端通过以上验收后，本文版本方可标记为生产可用。
