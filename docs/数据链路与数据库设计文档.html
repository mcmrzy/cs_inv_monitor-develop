2. 五条核心数据链路

链路 1：设备数据上报（Telemetry Ingest）

设备 MQTT→EMQX→Bridge→Kafka inv-telemetry→ProtocolParser→API Server HTTP→PG + Redis
MQTT Topic
cs_inv/{sn}/data/heartbeat — 3 分钟采样
有序消费
runOrderedKafkaConsumer 严格保序：FetchMessage → process → CommitMessages，指数退避（base 250ms, max 5s）
Heartbeat V1
6 个分组：ac(8) / bat(23) / pv(7) / sys(11) / eng(12) / cells(2)
协议解析
ParseRuleEngine 根据 device_protocol_fields 动态映射，范围校验 + NaN 过滤 + 时钟守卫
SHA256 去重
data_hash = SHA256(sn|event_time|seq)，ON CONFLICT DO NOTHING
唯一约束
PRIMARY KEY (device_sn, event_time)，唯一索引 uq(sn, event_time, data_hash)
时钟守卫
偏差 > ±5min → quality_flags |= 0x0004
串行锁
pg_advisory_xact_lock 保证同设备衍生写入串行
HTTP 内部调用
X-Internal-Key 头 → /api/v1/internal/telemetry-v2
quality_flags 位掩码: Bit0(0x01) OutOfRange | Bit1(0x02) NaN | Bit2(0x04) ClockInvalid | Bit3(0x08) Backfill

链路 2：API 请求（REST API）

前端/APP→Gateway :8888→API Server :8080→PG / Redis
三级路由
public(无认证) · user(JWT+RBAC) · admin(JWT+RBAC+role≤1)
JWT
HS256，与 EMQX 共享 Secret，注入 X-User-ID / X-User-Role
RBAC
三元组 (role, resource, action)，Redis TTL 300s
限流
login 10QPS · send-code 2QPS · 默认 50QPS
级别认证权限典型路径
Public无无/api/v1/auth/login, /register
UserJWTJWT+RBAC/api/v1/devices/*, /stations/*
AdminJWTrole≤1/api/v1/admin/users, /admin/models

链路 3：控制命令下发（Command & Control）

客户端→Gateway→API Server→Device Server→MQTT Pub → 设备
命令模型
device_model_commands: JSONB parameter_schema / response_schema, 风险 1~3
生命周期
device_commands: pending → queued → sent → acknowledged → completed/timeout
MQTT
Pub cs_inv/{sn}/cmd/{code} QoS 1
离线队列
Redis List device:cmd:queue:{sn}，上线 FlushPendingCommands
设备影子
device_control_state: desired/reported 双状态

链路 4：告警数据（Alert Pipeline）

设备 alarm→EMQX→Bridge→Kafka inv-alerts→AlertConsumer→API Server→不可变事件 → 可变投影
Topic
cs_inv/{sn}/data/alarm
格式
RawAlarmPayload{code, level, message, count, alarms[], timestamp}
去重
Redis alert:dedup:{MD5} TTL 60s
单事务双模型
先追加 device_alarm_events 不可变审计事件，再更新 alarms 当前生命周期/人工处理投影；两表不是等待合并的新旧表
投影列兼容
normalize_alarms_compat_columns() 仅同步 alarms 内的 alarm_level/level 等别名列，不负责跨表同步
快照
device_alarm_snapshots before/after 遥测快照

链路 5：OTA 升级

上传固件→API Server→firmware_versions
创建任务→API Server→MQTT→设备→上报进度→状态同步
固件
firmware_versions: UNIQUE(model, version), MD5
灰度
upgrade_packages: 4 种 (all/model/user/device)
状态机
upgrade_tasks: draft → pending → running → completed/failed
4 固件版本
devices: ARM/ESP/DSP/BMS

宽表字段晋升策略

标准且稳定的高频查询、连续聚合和告警字段保留类型化列。实验性、型号专属或低频字段放入受控 data.ext 对象，使用厂商命名空间，并在 telemetry_field_catalog 登记；完整值由 raw_envelope 保存，不因每个扩展字段执行 ALTER TABLE。只有形成稳定查询/聚合需求后，才通过新迁移晋升为类型化列并同步协议映射。

3.2 衍生数据与维护链

衍生表维护方式说明
device_latest_state触发器 maintain_telemetry_v2_derived()每设备一行最新快照，quality_flags&8 (Backfill) 时跳过
device_energy_day触发器 UPSERT日能量统计，取 GREATEST 保证单调递增
device_energy_month定时任务 refresh_device_energy_month()月能量统计，每日 2:10 CST 刷新，回溯 4 个月
device_cell_samples直接写入电芯采样 Hypertable，voltages/temperatures JSONB 数组
device_latest_cells触发器 maintain_latest_cells()每设备最新电芯数据
device_telemetry_hour连续聚合 Materialized View小时级聚合，10 分钟刷新，offset 2 天

衍生数据维护链流程图

device_telemetry_3min INSERT
  ├─ 触发器 maintain_telemetry_v2_derived()
  │    ├─ pg_advisory_xact_lock('telemetry:v1:' || device_sn)
  │    ├─ device_latest_state UPSERT (quality_flags&8==0 时)
  │    └─ device_energy_day UPSERT
  │              └─ 定时任务(每日 2:10 CST) → device_energy_month
  └─ 连续聚合 → device_telemetry_hour (10min 刷新)

device_cell_samples INSERT
  └─ 触发器 maintain_latest_cells()
       ├─ pg_advisory_xact_lock('cells:v1:' || device_sn)
       └─ device_latest_cells UPSERT
迁移 047 锁隔离：同一数据流、同一设备仍严格串行；遥测、电芯和 alarm-lifecycle:v1:{sn}:{source}:{code} 告警生命周期使用不同命名空间，不再因 SN 相同而跨流互相阻塞。

3.3 告警域

表类型说明
alarms可变业务投影 / Hypertable供告警列表、当前生命周期与确认/忽略等人工处理使用；生命周期字段可重建，人工处理字段需单独保全
device_alarm_events不可变事件日志 / 普通表active/recovered 完整信封审计，SHA256 data_hash 与事件幂等键；业务代码只追加不更新
device_alarm_snapshots普通表告警前后遥测快照 (before/after), FK → device_alarm_events
alert_rules普通表JSONB conditions, cooldown_minutes 1~1440, severity 分级
兼容触发器 normalize_alarms_compat_columns(): BEFORE INSERT/UPDATE 双向同步 alarm_level↔level, fault_message↔message, 校验 status IN (0,1,2) 和 event_state IN ('active','recovered')
一致性边界：IngestAlarmV1 在同一事务中以事件 INSERT 为幂等屏障，写快照并完成乱序判断后才更新 alarms 投影；任一步失败均回滚。生产路由不再编译或挂载旧 best-effort DeviceAlarm。

投影校验与重建计划

  1. 按 (device_sn, source, code, event_time, id) 重放事件；同时间戳 recovered 为终态，乱序旧事件不倒退投影。
  2. 先生成影子投影，对比 active/recovered 与发生/恢复时间；差异为零才在维护窗口切换。
  3. status/handled_at/handled_by 不从设备事件猜测，必须从原投影及操作审计保留；证据不足则人工复核。
  4. 每日只读检查最新事件/投影状态、活动唯一性和快照（含明确 missing）完整性；事件错误使用补偿事件，不原地修改。

7. 设计亮点与注意事项

设计亮点

#亮点说明
1配置驱动型号注册device_models + device_protocol_fields + device_model_fields 全配置化，零代码新型号接入
2触发器下沉衍生逻辑latest_state / energy_day 由 PostgreSQL 触发器维护，强一致性，应用层无需关心
3多层数据质量守卫范围校验 (protocol min/max) + 时钟守卫 (±5min) + NaN 过滤 + quality_flags 位掩码
4SHA256 去重 + 有序 Kafka 消费data_hash 幂等写入 + runOrderedKafkaConsumer 严格保序，数据不乱序不重复
5设备影子模型device_control_state desired/reported 双状态，IoT 标准模式，支持离线命令队列
6连续聚合device_telemetry_hour Materialized View 自动维护小时级聚合，Dashboard 查询毫秒级
7兼容视图桥接v_device_telemetry_compat 将宽表列重构为旧 JSONB 结构，读路径平滑迁移
8告警事件溯源 + 业务投影同事务先追加 device_alarm_events，再更新 alarms；事件负责审计与重放，投影负责列表和人工处理

注意事项

#事项说明
1旧 device_telemetry 表保留过渡期只读，v_device_telemetry_compat 视图已桥接到新表
2alarms 列别名兼容alarm_level/level、fault_message/message 通过触发器同步；这是单表列兼容，不是两张告警表的兼容双写
3宽表 Schema 变更成本已采用“稳定核心列 + raw_envelope + 字段目录”晋升策略；厂商扩展不再要求每次 ALTER TABLE
4同设备串行锁迁移 047 已按 telemetry/cells/alarm 生命周期划分命名空间，只保留同流同设备顺序；继续监控单设备异常突发造成的同流等待
5alarms_old 备份表迁移 036 保留 alarms_old 用于回滚安全，生产环境确认后可清理
6QualityBackfill 位quality_flags Bit3(0x08) 标记回填数据，触发器跳过 latest_state 更新
