# 48V 离网逆变器遥测协议与存储重设计

## 1. 文档目标

本文根据以下两部分重新设计设备遥测链路：

- `系统参数规范_48V离网逆变器.md` 中 CS-I10-6k2 的完整参数。
- 当前 Web 管理后台、Flutter App、API Server 和 Device Server 的实际功能及查询方式。

本次改造不只是替换 `device_telemetry` 表，而是同步调整：

- 设备上行 MQTT 协议。
- MQTT/Kafka 消息分类。
- Device Server 协议解析与标准字段映射。
- Redis 实时状态。
- PostgreSQL/TimescaleDB 时序存储与聚合。
- API Server 查询和统计逻辑。
- Web/App 数据展示、导出及测试。

基础设施统一使用 PostgreSQL 18、TimescaleDB 2.27.1 或更高兼容版本，以及 Redis 7，不引入 ClickHouse。生产镜像必须锁定经过验证的具体补丁版本，不能长期使用浮动的 `latest` 标签。

PostgreSQL 18 上线前必须验证 TimescaleDB 的 PG18 构建，并完成超表、连续聚合、压缩、保留策略、备份恢复和 `pg_upgrade` 演练。已有 PostgreSQL 16 数据卷不能直接挂载给 PostgreSQL 18 容器。

## 2. 功能需求与数据粒度

### 2.1 当前功能的数据用途

| 功能 | 数据范围 | 建议数据源 |
|---|---|---|
| 设备实时详情 | 最新完整快照 | Redis，PostgreSQL 最新状态兜底 |
| 设备列表、电站实时功率 | 所属设备最新值 | PostgreSQL 最新状态表 |
| 在线/离线判断 | 最后一次有效 heartbeat | Redis TTL |
| 实时曲线 | 1h、6h、24h、7d、自定义 | 3 分钟明细或小时聚合 |
| 历史数据列表 | 默认最近 7 天 | 小时聚合 |
| App 日曲线 | 指定某日 | 小时聚合 |
| App 月曲线 | 指定某月 | 日统计 |
| App 年/累计曲线 | 指定年份或全部 | 日/月统计 |
| 仪表盘趋势 | 7 日、28 日、12 个月 | 日/月统计 |
| 能源统计 | 今日、本月、今年、累计 | 日能源表 |
| 设备曲线对比 | 指定设备和时间范围 | 3 分钟/小时聚合 |
| 遥测导出 | 指定设备和时间范围 | 按跨度自动选择 3 分钟/小时/日 |
| 故障分析 | 告警前后遥测 | 告警事件和 3 分钟快照 |

### 2.2 数据分层原则

1. 实时查询不得扫描时序历史表。
2. 一天以上的页面查询不得读取并在 Go 内存中聚合全部 3 分钟记录。
3. 月、年和累计能源不得从 3 分钟表重复计算。
4. 设备端数组格式只存在于版本解析器中，不能泄漏到业务层和前端。
5. 高频快照、低频详情、事件和静态信息分别存储。

## 3. 数据量与保留边界

最大规划规模为 10 万台设备，每台每 3 分钟一个完整快照：

```text
100,000 / 180 = 556 条/秒
100,000 * 480 = 4,800 万条/天
100,000 * 175,200 = 175.2 亿条/年
```

因此，“3 分钟明细永久在线保留”仍与单机 PostgreSQL/TimescaleDB 的合理容量不匹配。业务上的“永久保留”应定义为统计结果和可追溯结果永久保留，而不是所有 3 分钟采样永久保持热查询。

推荐生命周期策略：

| 数据 | 开始压缩 | 自动删除 | 设计依据 |
|---|---:|---:|---|
| 最新状态 | 不压缩 | 不删除 | 每设备一行且持续 UPSERT |
| 3 分钟遥测 | 3 天 | 90 天 | 近期曲线、诊断和明细导出 |
| 小时聚合 | 30 天 | 不删除 | 跨月、跨年历史曲线 |
| 日统计 | 不压缩 | 不删除 | 月、年和累计业务统计依据 |
| 月统计 | 不压缩 | 不删除 | 长期趋势数据量很小 |
| 告警事件 | 90 天 | 不删除 | 近期记录需要确认、处理和更新 |
| 告警前后快照 | 30 天 | 不删除 | 永久故障追溯，写入后不再修改 |
| 正常电芯样本 | 7 天 | 90 天 | 近期电池诊断，单条数据较宽 |
| 异常电芯样本 | 30 天 | 不删除 | 与告警关联，永久追溯 |
| 设备控制日志 | 30 天 | 1,095 天 | 运维审计保留 3 年 |
| OTA 状态明细 | 30 天 | 1,095 天 | 升级追溯保留 3 年 |
| 原始 MQTT payload | 1 天 | 30 天 | 仅用于协议和解析问题排查 |

说明：

- “开始压缩 3 天”表示三天前的已关闭 chunk 转为压缩存储，不表示只保留三天。
- 最新状态、日表和月表数据量有限且经常更新，不应使用 TimescaleDB 压缩。
- 告警事件在确认、处理、恢复前可能更新，因此延迟到 90 天后压缩。
- 告警快照和异常电芯样本写入后应保持不可变，适合长期压缩保存。
- 控制日志和 OTA 明细默认保留 3 年；如合同或法规要求更长，应以合规期限覆盖该默认值。
- 删除 3 分钟数据前，必须确认对应小时聚合已经刷新完成。

### 3.1 三分钟数据容量保护

90 天是业务保留上限，不代表任何规模下都应无条件保持 90 天。10 万台全部在线时，90 天约产生 43.2 亿行 3 分钟数据，单集群承载成本仍然很高。系统应根据活跃设备数设置容量保护：

| 活跃设备数 | 建议 3 分钟明细保留 | 说明 |
|---:|---:|---|
| 不超过 5,000 | 90 天 | 可满足较长范围的精细诊断 |
| 5,001~20,000 | 60 天 | 以小时聚合承接长期查询 |
| 20,001~50,000 | 30 天 | 3 分钟数据只服务近期诊断 |
| 超过 50,000 | 14~30 天 | 根据实际压缩率和磁盘水位决定 |

生产环境不能仅依赖固定天数，还应监控：

- 时序库磁盘使用率达到 70% 时预警。
- 达到 80% 时停止延长 3 分钟明细保留期并评估提前清理。
- 达到 85% 时按最老 chunk 执行容量保护，但不得删除尚未完成小时聚合的数据。
- 每月根据活跃设备数、日增量、压缩率和剩余容量重新计算安全保留天数。

如果业务合同明确要求所有 3 分钟采样点永久保存，需要另行确定分片数量、磁盘预算、备份和冷存储方案。

## 4. MQTT 主题重新规划

### 4.1 上行主题

| 主题 | QoS | 频率 | 说明 |
|---|---:|---:|---|
| `cs_inv/{sn}/heartbeat` | 1 | 180s | 完整 3 分钟遥测及电芯快照 |
| `cs_inv/{sn}/info` | 1 | 连接成功、信息变化 | 设备与固件信息 |
| `cs_inv/{sn}/alarm` | 1 | 事件触发 | 告警发生和恢复 |
| `cs_inv/{sn}/config` | 1 | 连接、配置变化、查询后 | 设备实际生效配置 |
| `cs_inv/{sn}/parallel` | 0 | 5min、拓扑变化 | 并机拓扑和成员状态 |
| `cs_inv/{sn}/three_phase` | 0 | 180s | 三相系统数据，仅主机 |
| `cs_inv/{sn}/cmd/response` | 1 | 事件触发 | 控制命令回复 |
| `cs_inv/{sn}/ota/status` | 1 | OTA 期间 | OTA 状态 |
| `cs_inv/{sn}/status` | 1 | 上线/LWT | 仅表示网络在线状态 |

`heartbeat` 取代旧的：

- `data/ac`
- `data/battery`
- `data/pv`
- `data/status`
- `data/energy`

正常电芯快照合并到 heartbeat，不再使用独立的周期性 `cells` topic。告警、并机和 OTA 仍保持独立消息；电芯异常也必须立即通过 alarm topic 上报，不能等待下一次 heartbeat。

### 4.2 下行主题

| 主题 | QoS | 说明 |
|---|---:|---|
| `cs_inv/{sn}/cmd` | 1 | 设备控制与参数设置 |
| `cs_inv/{sn}/ota/cmd` | 1 | OTA 控制 |

### 5.2 顶层字段

| 字段 | 类型 | 必填 | 说明 |
|---|---|---:|---|
| `v` | uint16 | 是 | 协议版本，当前固定为 1 |
| `t` | int64 | 是 | 设备采样 Unix 秒，UTC |
| `seq` | uint32 | 是 | 设备递增序号，重启允许归零 |
| `ac` | array[8] | 是 | 交流输出参数 |
| `bat` | array[21] | 是 | BMS 参数 |
| `pv` | array[12] | 是 | 双路 MPPT 参数 |
| `sys` | array[10] | 是 | 系统状态 |
| `eng` | array[8] | 是 | 能源统计 |
| `cells` | array[2] | 是 | 电芯电压数组和温度数组 |

约束：

- 数组长度必须固定，不能省略尾部字段。
- 不支持或本次无效的值使用 `null`。
- `0` 是有效业务值，不能表示缺失。
- V1 下标含义永久不变；增加或调整字段必须发布 V2。
- SN 只从 MQTT topic 获取，payload 不重复发送 SN。
- `cells[0]` 为电芯电压数组，`cells[1]` 为电芯温度数组。
- 两个电芯数组长度必须等于设备 info 中登记的 `cell_count`。

## 6. 其他上行消息

### 6.1 设备信息

静态信息包含字符串且频率低，使用具名 JSON，不要求数组化：

```json
{
  "v": 1,
  "t": 1783676930,
  "model": "CS-I10-6k2",
  "manufacturer": "辰烁科技",
  "firmware_arm": "V1.2.3",
  "firmware_esp": "V1.0.5",
  "firmware_dsp": "V1.1.0",
  "firmware_bms": "V2.0.1",
  "rated_power": 6200,
  "rated_voltage": 220,
  "rated_frequency": 50.0,
  "battery_voltage": 51.2,
  "battery_type": "LiFePO4",
  "cell_count": 16,
  "parallel_mode": "standalone",
  "parallel_role": "standalone",
  "parallel_phase": null
}
```

### 6.2 电芯数据

电芯数据作为 heartbeat 的 `cells` 成员每 3 分钟上报一次，不再单独发布周期性 topic。传输层合并不代表存储层合并：后端解析后将普通遥测写入 3 分钟主表，将电芯数组写入电芯时序表，以便采用不同的压缩和删除策略。

两个内部数组长度必须与 `cell_count` 一致。电芯过压、欠压、温度或压差异常时，设备必须立即发布独立 alarm 事件，不能等待下一次 heartbeat。

### 6.3 告警事件

```json
{
  "v": 1,
  "t": 1783676930,
  "seq": 183,
  "source": 1,
  "code": 8,
  "level": 2,
  "state": 1
}
```

- `source`: 1 PCS、2 BMS、3 MPPT、4 COMM。
- `level`: 1 warning、2 fault。
- `state`: 1 active、0 recovered。
- 设备只发送稳定编码，名称、描述和多语言文本由后端字典映射。

### 6.4 并机与三相

并机成员列表包含 SN 和变长结构，继续使用具名 JSON：

- 拓扑变化时立即上报。
- 稳定状态每 5 分钟校准一次。
- 三相电压、电流、功率和不平衡度每 3 分钟上报。
- 并机拓扑保存最新状态及变更事件；三相指标进入单独 3 分钟超表。

## 7. CS-I10-6k2 控制与配置协议

### 7.1 设计原则

- 控制命令是低频关键操作，命令名保持可读，不为节省少量字节改成数字下标。
- 每条命令必须携带全局唯一 `task_id`，设备回复必须原样返回。
- “MQTT 已发送”“设备已收到”“执行成功”“配置已经生效”是四个不同状态。
- 命令执行结果保存在命令表；设备实际配置保存在配置状态表；二者不写入遥测历史。
- 参数在设备协议中使用整数缩放，避免不同语言的浮点误差。
- 控制接口必须按设备型号校验命令、范围、在线状态和用户权限。

### 7.2 下行命令格式

```json
{
  "v": 1,
  "task_id": "019f5b9e-7bd7-7e50-a14d-5c8a74235a10",
  "t": 1783676930,
  "cmd": "set_soc_window",
  "args": [200, 950]
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `v` | uint16 | 控制协议版本 |
| `task_id` | UUIDv7 string | 幂等、追踪和回复关联键 |
| `t` | int64 | 云端创建命令的 Unix 秒 |
| `cmd` | string | 稳定命令编码 |
| `args` | array | 按该命令固定顺序排列的参数 |

设备应保存最近一批已执行的 `task_id`。收到重复任务时不得重复执行，应返回上一次结果。

### 7.3 逆变器控制命令

| 命令 | args | 校验 | 影响的配置字段 |
|---|---|---|---|
| `ac_on` | `[]` | 设备在线且无禁止启动故障 | `ac_enabled=true` |
| `ac_off` | `[]` | 设备在线 | `ac_enabled=false` |
| `set_power_limit` | `[watts]` | 0~6200 W | `power_limit_w` |
| `set_charge_limit` | `[watts]` | 0~6200 W | `charge_limit_w` |
| `set_discharge_limit` | `[watts]` | 0~6200 W | `discharge_limit_w` |
| `set_soc_low` | `[soc_x10]` | 100~500，即 10.0%~50.0% | `soc_low_x10` |
| `set_soc_high` | `[soc_x10]` | 500~1000，即 50.0%~100.0% | `soc_high_x10` |
| `set_soc_window` | `[low_x10, high_x10]` | low < high，且间隔至少 5% | 两个 SOC 边界 |
| `force_charge` | `[enabled]` | 0/1，不得与强制放电同时开启 | `force_charge` |
| `force_discharge` | `[enabled]` | 0/1，不得与强制充电同时开启 | `force_discharge` |
| `restart` | `[]` | 故障复位权限 | 不改变持久配置 |
| `query_config` | `[]` | 设备在线 | 触发 config 上报 |
| `query_telemetry` | `[]` | 设备在线 | 触发一次 heartbeat |

平台还必须校验 `soc_low_x10 < soc_high_x10`。强制充放电属于临时运行指令，应支持超时自动取消，不能只依赖用户再次关闭。

### 7.4 BMS 控制命令

| 命令 | args | 校验 | 影响的配置字段 |
|---|---|---|---|
| `bms_set_enable` | `[charge, discharge]` | 每项为 0/1 | 两个使能状态 |
| `bms_set_charge_current` | `[amp_x10]` | 0~600，即 0~60.0 A | `max_charge_current_x10` |
| `bms_set_discharge_current` | `[amp_x10]` | 0~1200，即 0~120.0 A | `max_discharge_current_x10` |
| `bms_set_charge_voltage` | `[volt_x10]` | 440~584，即 44.0~58.4 V | `charge_voltage_x10` |
| `bms_set_discharge_voltage` | `[volt_x10]` | 400~500，即 40.0~50.0 V | `discharge_voltage_x10` |

BMS 参数还应同时受设备实时温度、故障、保护状态及电池类型约束。云端通过范围校验不代表设备可以跳过本地安全校验。

### 7.5 并机控制命令

| 命令 | args | 说明 |
|---|---|---|
| `parallel_enable` | `[mode]` | 1 单相并联、2 三相组网 |
| `parallel_set_role` | `[role, machine_id]` | role: 1 master、2 slave |
| `parallel_set_phase` | `[phase]` | 1 L1、2 L2、3 L3 |
| `parallel_disable` | `[]` | 恢复单机模式 |
| `parallel_sync_start` | `[]` | 仅主机可执行 |
| `parallel_sync_stop` | `[]` | 仅主机可执行 |

改变并机模式、角色和相位属于高风险操作。必须要求设备停机、检查拓扑冲突，并记录操作人；不能与普通功率参数使用相同权限级别。

### 7.6 命令回复

设备收到命令后先回复 ACK，执行结束后再回复最终结果，均使用 `cs_inv/{sn}/cmd/response`：

```json
{
  "v": 1,
  "task_id": "019f5b9e-7bd7-7e50-a14d-5c8a74235a10",
  "t": 1783676931,
  "cmd": "set_soc_window",
  "stage": "completed",
  "success": true,
  "code": "OK",
  "message": "",
  "data": []
}
```

`stage` 枚举：

- `acknowledged`: 已收到并通过基本格式校验。
- `executing`: 正在执行，仅长耗时操作需要。
- `completed`: 执行结束，以 `success` 判断结果。

建议结果码：`OK`、`INVALID_ARGS`、`UNSUPPORTED_CMD`、`DEVICE_BUSY`、`SAFETY_BLOCKED`、`FAULT_ACTIVE`、`EXECUTION_FAILED`、`DUPLICATE_TASK`。

### 7.7 实际配置上报

设备连接成功、配置变化、命令成功执行或收到 `query_config` 后发布：

```json
{
  "v": 1,
  "t": 1783676932,
  "rev": 42,
  "inv": [1, 6200, 6200, 6200, 200, 950, 0, 0],
  "bms": [1, 1, 500, 1000, 584, 430],
  "parallel": [0, 0, 0, null]
}
```

`inv` 固定顺序：

| 下标 | 字段 | 单位/枚举 |
|---:|---|---|
| 0 | `ac_enabled` | 0/1 |
| 1 | `power_limit_w` | W |
| 2 | `charge_limit_w` | W |
| 3 | `discharge_limit_w` | W |
| 4 | `soc_low_x10` | 0.1% |
| 5 | `soc_high_x10` | 0.1% |
| 6 | `force_charge` | 0/1 |
| 7 | `force_discharge` | 0/1 |

`bms` 固定顺序：

| 下标 | 字段 | 单位/枚举 |
|---:|---|---|
| 0 | `charge_enabled` | 0/1 |
| 1 | `discharge_enabled` | 0/1 |
| 2 | `max_charge_current_x10` | 0.1 A |
| 3 | `max_discharge_current_x10` | 0.1 A |
| 4 | `charge_voltage_x10` | 0.1 V |
| 5 | `discharge_voltage_x10` | 0.1 V |

`parallel` 固定顺序为 mode、role、machine_id、phase。`rev` 是设备配置修订号，每次实际配置变化递增。

### 7.8 期望配置与实际配置

平台下发配置命令时先更新 desired；只有设备执行成功并重新上报 config 后才更新 reported。两者一致时配置同步状态为 synced。

```text
用户提交
  -> 校验权限、型号、范围和安全互锁
  -> 创建 device_commands(pending)
  -> 更新 device_control_state.desired
  -> MQTT 下发
  -> 设备 ACK
  -> 设备执行并回复 completed
  -> 设备发布 config
  -> 更新 reported
  -> 对比 desired/reported
  -> 标记 synced 或 drifted
```

命令超时或执行失败时不得把 desired 自动当成设备实际值。

## 8. 后端标准模型

### 8.1 运行时标准模型

协议解析器将数组转换为标准结构，后续代码不得读取数组下标：

```go
type TelemetrySample struct {
    ProtocolVersion uint16
    DeviceSN        string
    Sequence        uint32
    EventTime       time.Time
    ReceivedAt      time.Time
    QualityFlags    uint32

    AC      ACTelemetry
    Battery BatteryTelemetry
    PV      PVTelemetry
    System  SystemTelemetry
    Energy  EnergyTelemetry
}
```

目录建议：

```text
inv_device_server/internal/telemetry/
├── model.go
├── validator.go
├── quality.go
└── heartbeat/
    ├── parser.go
    ├── v1.go
    └── v1_test.go
```

协议版本切换规则：

- V1 解析器只负责 V1。
- 新版本新增文件和测试，不能修改已有下标含义。
- 未支持版本进入死信/错误日志，不得按最新版本猜测解析。

### 8.2 型号管理职责拆分

当前型号配置同时承担字段定义、协议解析、页面展示和控制参数，职责过多。重构后拆成四层：

| 层级 | 负责内容 | 谁可以修改 |
|---|---|---|
| 标准字段目录 | 全系统字段语义、基础单位和数据类型 | 系统研发 |
| 协议版本映射 | heartbeat 数组下标、wire type、缩放和范围 | 协议研发，版本发布 |
| 型号字段能力 | 型号支持哪些字段、显示、排序、曲线和告警 | 产品/型号管理员 |
| 型号控制能力 | 支持哪些命令、参数范围、风险和权限 | 产品/运维管理员 |

管理后台不得直接修改已经发布协议的数组下标、wire type、基础单位和缩放规则。

### 8.3 标准字段目录 `telemetry_field_catalog`

```sql
CREATE TABLE telemetry_field_catalog (
    field_key          VARCHAR(64) PRIMARY KEY,
    field_type         VARCHAR(20) NOT NULL,
    base_unit          VARCHAR(20),
    category           VARCHAR(32) NOT NULL,
    description        TEXT,
    is_timeseries      BOOLEAN NOT NULL DEFAULT TRUE,
    is_aggregatable    BOOLEAN NOT NULL DEFAULT TRUE,
    allowed_aggregates JSONB NOT NULL DEFAULT '[]',
    status             VARCHAR(20) NOT NULL DEFAULT 'active',
    created_at         TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at         TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

标准字段示例：

```text
ac_voltage
ac_active_power
battery_soc
battery_current
pv_total_power
inverter_temperature
daily_pv_energy
```

同一个 `field_key` 在所有型号和协议版本中必须保持相同业务含义、方向和基础单位。例如 `battery_current` 永远使用 A，正数充电、负数放电。

字段删除采用 `status=deprecated`，不能立即删除目录记录，也不能把旧 key 重新分配给其他含义。

### 8.4 协议版本与字段映射

```sql
CREATE TABLE device_protocol_versions (
    id                BIGSERIAL PRIMARY KEY,
    protocol_code     VARCHAR(64) NOT NULL,
    version           SMALLINT NOT NULL,
    schema_hash       VARCHAR(64) NOT NULL,
    status            VARCHAR(20) NOT NULL DEFAULT 'draft',
    released_at       TIMESTAMPTZ,
    created_at        TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (protocol_code, version)
);

CREATE TABLE device_protocol_fields (
    id                  BIGSERIAL PRIMARY KEY,
    protocol_version_id BIGINT NOT NULL REFERENCES device_protocol_versions(id),
    group_code          VARCHAR(16) NOT NULL,
    field_index         SMALLINT NOT NULL,
    field_key           VARCHAR(64) NOT NULL REFERENCES telemetry_field_catalog(field_key),
    wire_type           VARCHAR(20) NOT NULL,
    scale               NUMERIC NOT NULL DEFAULT 1,
    minimum             NUMERIC,
    maximum             NUMERIC,
    nullable            BOOLEAN NOT NULL DEFAULT TRUE,
    status              VARCHAR(20) NOT NULL DEFAULT 'active',
    UNIQUE (protocol_version_id, group_code, field_index)
);
```

Heartbeat V1 示例：

```text
heartbeat-v1, ac, 0   -> ac_voltage
heartbeat-v1, bat, 3  -> battery_current
heartbeat-v1, pv, 10  -> pv_total_power
heartbeat-v1, sys, 3  -> inverter_temperature
```

版本状态：`draft`、`released`、`retired`。只有 draft 可以编辑；released 只读。发布时计算 `schema_hash`，代码内 V1 映射和数据库镜像必须一致，否则部署检查失败。

协议映射以代码解析器为运行时权威，数据库用于审计、文档、后台只读展示和自动化一致性测试，不能让后台热修改运行中的解析行为。

### 8.5 型号基本信息

`device_models` 只保存型号本身及协议绑定，不再重复保存 `data_fields`、`field_mapping` 和 `mqtt_topics` 三套 JSON：

```sql
CREATE TABLE device_models (
    id                    BIGSERIAL PRIMARY KEY,
    model_code            VARCHAR(50) NOT NULL UNIQUE,
    model_name            VARCHAR(100) NOT NULL,
    manufacturer          VARCHAR(100) NOT NULL,
    category              VARCHAR(32) NOT NULL,
    heartbeat_protocol_id BIGINT REFERENCES device_protocol_versions(id),
    rated_power_w         INTEGER,
    rated_voltage_v       NUMERIC(8,2),
    rated_frequency_hz    NUMERIC(6,2),
    battery_voltage_v     NUMERIC(8,2),
    battery_type          VARCHAR(32),
    cell_count            SMALLINT,
    mppt_count            SMALLINT,
    supports_parallel     BOOLEAN NOT NULL DEFAULT FALSE,
    specifications        JSONB NOT NULL DEFAULT '{}',
    status                VARCHAR(20) NOT NULL DEFAULT 'draft',
    created_at            TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at            TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

核心规格使用类型化列，少量非通用规格放 `specifications`。型号状态固定为 `draft`、`active`、`disabled`、`retired`。已有设备引用的型号不能物理删除。

### 8.6 型号字段能力 `device_model_fields`

```sql
CREATE TABLE device_model_fields (
    id                  BIGSERIAL PRIMARY KEY,
    model_id            BIGINT NOT NULL REFERENCES device_models(id),
    field_key           VARCHAR(64) NOT NULL REFERENCES telemetry_field_catalog(field_key),
    display_name_key    VARCHAR(128),
    group_code          VARCHAR(32) NOT NULL,
    display_unit        VARCHAR(20),
    decimal_places      SMALLINT NOT NULL DEFAULT 1,
    sort_order          INTEGER NOT NULL DEFAULT 0,
    is_supported        BOOLEAN NOT NULL DEFAULT TRUE,
    is_visible          BOOLEAN NOT NULL DEFAULT TRUE,
    show_realtime       BOOLEAN NOT NULL DEFAULT TRUE,
    show_history        BOOLEAN NOT NULL DEFAULT TRUE,
    allow_compare       BOOLEAN NOT NULL DEFAULT FALSE,
    allow_alarm_rule    BOOLEAN NOT NULL DEFAULT FALSE,
    default_chart       BOOLEAN NOT NULL DEFAULT FALSE,
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    UNIQUE (model_id, field_key)
);
```

这张表是型号管理后台“监测字段”页面的主要数据源。它只决定该型号支持什么以及如何展示，不决定设备数组如何解析。

例如 CS-I10-6k2 支持 PV2 和 16 节电芯，单 MPPT 型号可以把 PV2 字段标记为不支持；前端据此隐藏字段，无需修改页面代码。

### 8.7 扩展字段机制

稳定、高频、参与统计的指标进入 heartbeat 固定数组和类型化数据库列。实验性、型号专属且低频的指标可以使用可选扩展区：

```json
"x": [[1001, 523.4], [1002, 1]]
```

扩展字段 ID 由 `telemetry_extension_fields` 统一注册，禁止复用。扩展字段确认需要长期统计后，再通过新协议版本升级为核心字段。扩展区不能替代正常的协议版本管理，也不能承载高频核心指标。

### 8.8 型号管理后台页面

型号管理重新拆成五个页签：

1. **基本信息**：型号编码、名称、制造商、额定参数、电池和并机能力。
2. **监测字段**：支持、显示、分组、排序、精度、曲线、对比和告警规则能力。
3. **控制能力**：命令启停、参数范围、风险等级、权限和超时。
4. **协议版本**：绑定的 heartbeat 版本、数组映射和 schema hash，只读展示已发布版本。
5. **数据预览**：使用样例 payload 解析并显示标准字段、无效字段和质量标记。

删除当前页面中允许业务管理员任意选择 JSON/Modbus/custom 解析器及编辑线上 parse config 的入口。协议草稿编辑应放在独立的研发权限页面，并通过发布流程生效。

页面不能硬编码 AC、PV、BMS 等字段列表，应由标准字段目录和型号字段绑定动态生成。分组使用稳定 `group_code`，显示名称通过国际化 key 翻译，不能把中文分组名直接作为数据库枚举。

### 8.10 权限与发布治理

| 操作 | 建议权限 |
|---|---|
| 查看型号和字段 | 型号查看权限 |
| 编辑显示、排序和曲线能力 | 型号编辑权限 |
| 编辑控制范围和风险 | 控制策略管理权限 |
| 创建协议草稿 | 协议研发权限 |
| 发布协议版本 | 协议发布权限，双人复核 |
| 停用/退役型号 | 型号管理员权限，影响检查 |

协议发布和高风险控制能力变更必须写入审计日志。已发布协议、已有设备型号和已被命令日志引用的 command code 均禁止硬删除。

### 8.11 旧型号数据迁移

当前表迁移原则：

1. 从 `device_model_field` 去重生成 `telemetry_field_catalog`。
2. 将每个型号现有字段迁移为 `device_model_fields` 绑定。
3. 将 `is_control/control_params` 转换为 `device_model_commands`，无法自动转换的复杂命令进入人工审核清单。
4. 根据 Heartbeat V1 文档生成只读 `device_protocol_versions/device_protocol_fields`。
5. 迁移期间先以临时新表构建并校验，切换后仍使用正式表名 `device_models`；旧表中的 `data_fields/field_mapping/mqtt_topics` 标记废弃并只读保留。
6. `device_model_protocol` 不再参与 heartbeat 运行时解析，历史 Modbus 型号继续保留专用适配器。
7. Web/App 全部切到新 API 后，再在后续版本删除旧编辑入口和废弃列。

迁移不能直接删除旧配置。必须生成新旧字段数量、field key、控制范围和协议映射对比报告，人工确认后激活新型号版本。

## 9. 数据库设计

### 9.1 `device_latest_state`

普通 PostgreSQL 表，每设备一行。保存 heartbeat 的全部标准字段以及：

- `device_sn` 主键。
- `protocol_version`。
- `sequence`。
- `event_time`。
- `received_at`。
- `quality_flags`。

用途：

- 设备实时详情兜底。
- 设备列表。
- 电站当前功率聚合。
- 仪表盘和大屏实时值。

更新条件必须比较采集时间和序号，离线补传不能覆盖最新状态。

### 9.2 `device_telemetry_3min`

TimescaleDB 超表，一台设备每 3 分钟一个逻辑样本。字段与标准模型一一对应，不保存 topic，不保存 JSONB，不使用自增 ID。电芯数组从标准样本拆分写入 `device_cell_samples`，不直接放入该宽表。

建议唯一键：

```sql
PRIMARY KEY (device_sn, event_time)
```

建议超表和压缩策略：

```sql
SELECT create_hypertable(
    'device_telemetry_3min',
    'event_time',
    chunk_time_interval => INTERVAL '1 day',
    if_not_exists => TRUE
);

ALTER TABLE device_telemetry_3min SET (
    timescaledb.compress,
    timescaledb.compress_segmentby = 'device_sn',
    timescaledb.compress_orderby = 'event_time DESC'
);

SELECT add_compression_policy(
    'device_telemetry_3min',
    INTERVAL '3 days',
    if_not_exists => TRUE
);

SELECT add_retention_policy(
    'device_telemetry_3min',
    INTERVAL '90 days',
    if_not_exists => TRUE
);
```

查询索引只保留：

```sql
CREATE INDEX ON device_telemetry_3min (device_sn, event_time DESC);
```

避免创建全字段索引、全局时间索引和 JSONB GIN 索引。

### 9.3 `device_telemetry_hour`

TimescaleDB 连续聚合，永久保留。每设备每小时一行，保存：

- 功率、电压、电流、SOC、温度的平均/最小/最大值。
- 状态和累计能源的最后值。
- 该小时有效样本数、在线分钟数和质量标记汇总。在线分钟数按有效采样数乘以 3 计算，每小时最多 60 分钟。

一天以上的历史曲线和导出优先读取小时表。

### 9.4 `device_energy_day`

每设备每天一行，永久保留：

- 当日发电、充电、放电和负载用电量。
- 四类累计能源日末值。
- 最大 PV、AC、充电和放电功率。
- SOC 平均值、最小值和最大值。
- 逆变器、MOS、电池最高温度。
- 在线分钟数、运行分钟数。
- 告警数、故障数和质量状态。
- `stat_date` 和计算使用的电站时区。

主键：

```sql
PRIMARY KEY (device_sn, stat_date)
```

月、年和累计统计从日表计算，不扫描 3 分钟明细表。

### 9.5 `device_energy_month`

每设备每月一行，永久保留。用于 App 年/累计曲线、后台 12 个月趋势和跨年查询。

可以由定时任务在日表基础上生成，也可以使用 TimescaleDB 连续聚合。由于涉及电站时区和月边界，推荐由可重算的日结任务生成。

### 9.6 控制能力表 `device_model_commands`

保存某型号支持哪些命令及其校验规则：

```sql
CREATE TABLE device_model_commands (
    id                  BIGSERIAL PRIMARY KEY,
    model_id            BIGINT NOT NULL,
    command_code        VARCHAR(64) NOT NULL,
    display_name        VARCHAR(100) NOT NULL,
    parameter_schema    JSONB NOT NULL DEFAULT '{}',
    response_schema     JSONB NOT NULL DEFAULT '{}',
    timeout_seconds     INTEGER NOT NULL DEFAULT 30,
    risk_level          SMALLINT NOT NULL DEFAULT 1,
    requires_online     BOOLEAN NOT NULL DEFAULT TRUE,
    is_enabled          BOOLEAN NOT NULL DEFAULT TRUE,
    UNIQUE (model_id, command_code)
);
```

`parameter_schema` 用于描述参数下标、类型、范围、缩放和互锁关系。控制协议是低频且不同型号差异明显，这里使用 JSONB 比不断增加固定列更合适。

### 9.7 命令生命周期表 `device_commands`

```sql
CREATE TABLE device_commands (
    id                  BIGSERIAL PRIMARY KEY,
    task_id             UUID NOT NULL UNIQUE,
    device_sn           VARCHAR(50) NOT NULL,
    command_code        VARCHAR(64) NOT NULL,
    requested_args      JSONB NOT NULL DEFAULT '[]',
    response_data       JSONB NOT NULL DEFAULT '[]',
    status              VARCHAR(20) NOT NULL DEFAULT 'pending',
    result_code         VARCHAR(64),
    result_message      TEXT,
    requested_by        BIGINT,
    source              VARCHAR(20) NOT NULL DEFAULT 'web',
    created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    queued_at           TIMESTAMPTZ,
    sent_at             TIMESTAMPTZ,
    acknowledged_at     TIMESTAMPTZ,
    completed_at        TIMESTAMPTZ,
    timeout_at          TIMESTAMPTZ NOT NULL,
    retry_count         SMALLINT NOT NULL DEFAULT 0
);

CREATE INDEX idx_device_commands_sn_created
    ON device_commands (device_sn, created_at DESC);
CREATE INDEX idx_device_commands_pending
    ON device_commands (status, timeout_at)
    WHERE status IN ('pending', 'queued', 'sent', 'acknowledged', 'executing');
```

状态固定为 `pending`、`queued`、`sent`、`acknowledged`、`executing`、`success`、`failed`、`timeout`、`cancelled`。服务端只能按合法状态机向前更新，迟到回复可以记录，但不能把已取消任务恢复成执行中。

### 9.8 配置状态表 `device_control_state`

```sql
CREATE TABLE device_control_state (
    device_sn           VARCHAR(50) PRIMARY KEY,
    protocol_version    SMALLINT NOT NULL DEFAULT 1,
    desired             JSONB NOT NULL DEFAULT '{}',
    reported            JSONB NOT NULL DEFAULT '{}',
    desired_version     BIGINT NOT NULL DEFAULT 0,
    reported_revision   BIGINT NOT NULL DEFAULT 0,
    sync_status         VARCHAR(20) NOT NULL DEFAULT 'unknown',
    desired_at          TIMESTAMPTZ,
    reported_at         TIMESTAMPTZ,
    last_task_id        UUID,
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

`sync_status` 固定为 `unknown`、`pending`、`synced`、`drifted`、`failed`。desired/reported 在 API 中转换为具名结构，数据库 JSONB key 使用本文定义的标准字段名，不保存设备端数组下标。

不建议把实际配置的每次变化覆盖后就完全丢失。每次成功变化还应写入 `device_control_events`，记录旧值、新值、task_id、操作人和时间，用于审计及问题追溯。

### 9.9 其他表

- `device_cell_samples`: 从 heartbeat 拆分写入的 3 分钟电芯时序超表。
- `device_latest_cells`: 每设备最新电芯快照。
- `device_alarm_events`: 告警发生和恢复事件。
- `device_alarm_snapshots`: 告警前后关键遥测快照。
- `device_parallel_topology`: 当前并机拓扑。
- `device_parallel_events`: 并机拓扑变更历史。
- `device_three_phase_3min`: 三相 3 分钟指标超表。
- `device_control_events`: 控制配置变更历史，保留 3 年。
- 设备与固件信息继续保存到 `devices` 等关系表，不进入遥测历史。

## 10. Redis 设计

建议键：

| Key | 值 | TTL |
|---|---|---:|
| `device:heartbeat:{sn}` | 最近接收 Unix 时间 | 600s |
| `device:latest:{sn}` | 标准具名 JSON | 600s |
| `device:cells:{sn}` | 从 heartbeat 提取的最新电芯 JSON | 600s |
| `device:online:{sn}` | 在线状态辅助信息 | 600s |

Redis 中不得保存设备端定长数组。WebSocket、SSE 和内部 API 统一读取标准具名字段。

在线规则：

- 实时 heartbeat 才刷新 TTL。
- 超过 24 小时的离线补传只写历史，不刷新在线状态。
- LWT 可立即标记离线，TTL 作为异常断线兜底。

## 11. 校验、幂等与质量标记

### 11.1 协议错误

以下情况拒绝整条消息：

- 缺少 `v`、`t` 或 `seq`。
- 不支持的协议版本。
- 任一固定数组长度错误。
- 数组元素类型无法解析。
- topic 中 SN 不合法。

### 11.2 业务数据异常

单字段越界时不拒绝整条消息：

- 异常字段转换为 null。
- 其余有效字段继续写入。
- 设置 `quality_flags`。
- 记录按型号和字段统计的异常指标。

建议质量位：

| 位值 | 含义 |
|---:|---|
| 1 | 设备时钟异常 |
| 2 | 字段数值越界 |
| 4 | 重复消息 |
| 8 | 乱序消息 |
| 16 | 字段计算不一致 |
| 32 | 部分字段缺失 |
| 64 | 离线补传 |

一致性校验：

- `pv_total_power` 与 `pv1_power + pv2_power` 比较。
- 电池功率符号与充放电状态比较。
- AC 视在功率与电压、电流比较。
- 功率因数不得超过 1。
- 累计能源原则上不能回退；设备更换或复位必须记录基线事件。

### 11.3 幂等规则

- 以 `(device_sn, event_time)` 作为 3 分钟样本幂等键。
- 设备采样时间应对齐到 3 分钟边界；同一时间槽多条消息优先保留采集时间更新、序号更新且质量更高的样本。
- `seq` 用于识别重复、乱序和掉包，不单独作为数据库主键。

## 12. API 查询规划

### 12.1 实时接口

```text
GET /devices/:sn/realtime
```

读取顺序：Redis `device:latest:{sn}`，失败时查询 `device_latest_state`。

### 12.2 历史曲线

```text
GET /devices/:sn/telemetry
```

根据时间跨度选择数据源：

| 查询跨度 | 数据源 | 返回粒度 |
|---|---|---|
| 不超过 48 小时 | 3 分钟表 | 3 分钟 |
| 2~90 天 | 小时表 | 小时 |
| 超过 90 天 | 日/月表 | 日或月 |

API 必须限制最大返回点数，例如 2,000 点，并根据跨度自动降采样。

### 12.3 能源与统计

- 今日实时能源可读取最新状态中的 daily 值。
- 历史日、月、年、累计能源读取日/月表。
- 电站统计按所属设备的日表聚合。
- 禁止在 Handler 中编写 JSONB 解析 SQL。

### 12.4 导出

- 导出任务必须分页或流式输出，不能一次加载到内存。
- 大范围导出自动使用小时或日数据。
- 如果用户明确选择 3 分钟明细数据，限制在明细表保留期内。

### 12.5 控制与配置接口

```text
GET  /devices/:sn/commands
POST /devices/:sn/commands
GET  /devices/:sn/commands/:task_id
GET  /devices/:sn/control-state
POST /devices/:sn/control-state/sync
```

- 创建命令接口接收具名业务参数，由 API Server 转换为设备 `args` 数组。
- 前端不得提交设备数组下标，也不得自行执行缩放换算。
- `control-state` 同时返回 desired、reported、差异字段和同步状态。
- 高风险并机命令需要更高权限和二次确认，并写入用户操作审计。
- 批量控制为每台设备创建独立 task_id，不能用一条命令日志代表整批设备。

## 13. 服务改造范围

### 13.1 `inv_device_server`

- 新增 heartbeat V1 DTO、解析器、校验器和标准模型。
- heartbeat 到达后更新 Redis TTL 和最新值。
- UPSERT `device_latest_state`。
- 将标准样本写入批量队列，再分别批量写 3 分钟遥测超表和电芯时序表。
- 故障码变化触发状态机，但告警生命周期以 alarm topic 为准。
- 删除旧五类 topic 拼接和旧 `device_telemetry` 逐 topic 写入。
- 离线补传只写历史，不更新实时状态。
- 订阅 config 和 cmd/response，按 task_id 转发 ACK、最终结果和实际配置。
- 对相同 task_id 的重复回复实施幂等处理。
- 运行时按设备型号绑定的 released 协议版本选择代码解析器。
- 启动和元数据刷新时校验代码 schema hash 与数据库协议镜像一致。
- 不再从业务管理员可编辑的 `parse_config` 热加载 heartbeat 数组下标。

### 13.2 `mqtt-kafka-bridge`

- heartbeat 转发到独立 Kafka topic。
- Kafka message key 使用设备 SN，保证同设备消息有序。
- Bridge 只做 topic 校验和可靠转发，不做业务字段映射。
- 若 Kafka 在生产中未启用，不应为了本次改造强制引入。

### 13.3 `inv_api_server`

- 实时详情改读 Redis/最新状态表。
- 历史接口按跨度选择 3 分钟、小时、日或月表。
- 仪表盘、电站统计、能源流、设备对比删除 JSONB 查询。
- 删除 `topic = 'data/...'` 过滤条件。
- 删除 Go 内存中的多 topic 时间槽拼接。
- 保持现有 API 对外字段名，降低 Web/App 改造范围。
- 使用 `device_model_commands` 完成型号、参数、风险等级和安全互锁校验。
- 命令发送前写入 `device_commands`，按回复推进状态机并处理超时。
- 收到 config 后更新 reported、计算 desired 差异并写配置变更事件。
- 新增标准字段目录、型号字段能力、型号命令和协议只读查询 API。
- 型号激活前执行完整性校验，已有设备引用的型号只能停用或退役，不能删除。
- 型号批量字段更新使用事务和乐观锁，并记录变更审计。

### 13.4 Web 管理后台

- 实时页面继续消费具名字段。
- 曲线页面允许 1h、6h、24h、7d、自定义范围。
- 显示后端实际返回的粒度。
- 长时间范围不应请求 3 分钟明细点。
- 导出界面提示所选粒度和可导出的时间范围。
- 控制页面展示设备实际值、待同步值和命令执行阶段，不能在 MQTT 已发送时提前显示成功。
- 并机和故障复位等高风险操作提供明确确认流程。
- 型号管理改为基本信息、监测字段、控制能力、协议版本和数据预览五个页签。
- 移除普通管理员编辑协议解析类型、数组下标和线上 parse config 的入口。
- 字段分组、显示、曲线和控制表单由新型号 API 动态生成。

### 13.5 Flutter App

- MQTT heartbeat 数组不能由 UI 页面直接解析。
- App 若继续直连 MQTT，应增加与后端相同的 V1 映射层。
- 日曲线使用小时数据，月曲线使用日数据，年/累计使用月数据。
- 实时页面内部统一使用具名实体。
- 控制参数由 API 返回的型号能力动态生成，App 不硬编码设备数组下标和范围。
- 实时字段、历史指标和单位由型号字段能力生成，型号不支持的字段不显示。

### 13.6 本次落地清单

数据库迁移按依赖顺序执行 `023_telemetry_v2_and_model_registry` 和
`024_model_registry_governance`。`024` 将旧结构字段和 `data_fields` 提升到标准字段目录，
生成每个型号的迁移报告；旧 `field_mapping` 只进入待人工复核报告，不自动推断为已发布的数组协议。

当前型号治理 API 如下：

| 域 | API | 权限 |

| 标准字段字典 | `GET/POST /field-catalog` | `models:view` / `models:dictionary` |

## 14. 切换方案

业务选择直接切换新表，不进行长期双写，但生产过程仍要保留回滚能力。

1. 创建新表、超表、聚合视图和监控指标。
2. 部署支持 heartbeat 的后端，但暂不停止旧协议。
3. 使用测试设备验证数组映射、符号、时间、序号和告警恢复。
4. 修改 API 查询到新表并完成接口回归测试。
5. 发布正式设备固件并启用 heartbeat 消费。
6. 停止旧五类 topic 写入。
7. 对比最新状态、3 分钟曲线、日能源、电芯和告警数据。
8. 将旧 `device_telemetry` 设置为只读归档。
9. 稳定运行并完成备份后，再删除旧视图、旧查询和旧表。

禁止在切换当天直接删除旧表。

## 16. PostgreSQL 职责下沉计划

### 16.1 结论

需要下沉，但只下沉由标准化类型列能够确定、没有外部副作用、可以在同一数据库事务中完成的逻辑。
MQTT 消费、heartbeat 协议解析、Redis 在线状态、告警生命周期、控制命令、OTA 和通知仍由后端负责。

| 能力 | 归属 | 判断 |
|---|---|---|
| 3 分钟事实数据和电芯事实数据写入 | Device Server | 后端负责解析、校验和批量写入 |
| 最新遥测状态 | PostgreSQL 触发器 | 只依赖新插入的类型化事实行，可原子维护 |
| 最新电芯状态 | PostgreSQL 触发器 | 只依赖电芯事实行，可原子维护 |
| 设备日能源汇总 | PostgreSQL 触发器 | 当前由每帧 Go SQL 维护，下沉后消除重复实现 |
| 设备月能源汇总 | TimescaleDB job | 每天 02:10 从永久日表幂等回算四个月，允许补传修正 |
| 小时遥测 | TimescaleDB continuous aggregate | 已由数据库承担 |
| 压缩和 90 天明细保留 | TimescaleDB policy | 已由数据库承担 |
| 离线判断 | API/Device Server + Redis | 涉及连接状态和事件通知，不下沉 |
| 告警、控制、OTA 状态机 | 后端服务 | 涉及权限、外部消息和重试，不下沉 |

### 16.2 实施步骤

1. 新增 `025_postgres_derived_telemetry` 迁移，创建遥测和电芯派生维护触发器。
2. 将日能源 UPSERT 从 Device Server 移入数据库函数，并保持乱序、补传和幂等语义。
3. 创建可手动重跑的月汇总过程，并注册每天 02:10 的 TimescaleDB job，默认回算最近四个月。
4. 精简 `SaveTelemetryV2`，仅在一个事务内插入 3 分钟事实行和电芯事实行。
5. 在 PostgreSQL 18 测试库验证新行、旧行、重复行、乱序行、补传和月汇总。
6. 执行 Go 全量测试并审查锁竞争、写放大、回滚和无 TimescaleDB 环境行为。

### 16.3 约束与回滚

- 触发器不得解析设备原始 JSON，只处理已经通过协议校验的类型化列。
- 基础事实表写入失败时，派生表更新必须同时回滚。
- 最新状态只接受时间更新的非补传数据；补传允许修正对应日/月统计。
- 月汇总过程必须幂等，重复执行得到相同结果。
- TimescaleDB job 注册失败不得影响事实数据写入，运维可手动调用月汇总过程补算。
- 回滚时可禁用两个触发器并恢复 Device Server 旧派生 SQL；事实数据格式不变。

### 16.4 性能验收记录

在 PostgreSQL 18.4、TimescaleDB 2.28.2 的隔离容器中使用 10 客户端、4 线程执行完整事务，
每个事务写入一行 3 分钟遥测和一行电芯数据，并触发最新遥测、日汇总和最新电芯维护：

- 15 秒完成 17,166 个事务。
- 0 个失败事务。
- 平均延迟 8.738ms。
- 吞吐 1,144.39 个 heartbeat 事务/秒。

10 万设备每 180 秒一次的平均入口需求约为 556 个 heartbeat/秒，本次隔离测试约有 2.06 倍余量。
生产仍需监控数据库 CPU、WAL、锁等待、Kafka backlog 和派生触发器延迟；若持续入口超过
800 个/秒或余量低于 30%，下一阶段应改为 COPY 批量事实写入加批量派生过程，而不是增加行级触发器逻辑。
