## 1. 概述

本次重构基于设计文档《辰烁科技 48V 离网逆变器控制参数与三级权限设计 V1.0》，将原有的粗粒度 `devices:control` 单一权限升级为 8 个细粒度权限码，并引入 schema 驱动的动态表单、动态限值校验引擎、9 步命令校验链、电池模板管理、能源计划与临时覆盖等完整能力。

采用**渐进式部署策略**，分三批推进：

| 批次 | 名称 | 步骤范围 | 目标 |
|------|------|---------|------|
| **P0-A** | 安全堵漏 | 步骤 1–5 | 消除 systemCommands 白名单旁路，实现 fail-closed，控制权限与查看权限分离 |
| **P0-B** | 细粒度权限与表单 | 步骤 6–11 | 命令级 permission_code 校验、schema 驱动的动态表单生成 |
| **P1** | 功能扩展 | 步骤 12–21 | 电池模板、能源计划、动态限值校验、9 步校验链、Admin 五页重构 |

## 3. 新增数据模型

### 3.1 迁移 048 — 系统命令注册与控制权限授予

**目标**：为移除白名单做数据准备，将历史硬编码的 6 个系统命令持久化到 `device_model_commands`。

| 操作类型 | 对象 | 说明 |
|---------|------|------|
| INSERT | `device_model_commands` | 为 `CS-I10-6k2` 注册 6 个命令：`get_params`（risk 1）、`set_params`（risk 2）、`set_control`（risk 2）、`set_alarm`（risk 2）、`batch_config`（risk 2）、`reset`（risk 2） |
| INSERT | `role_permissions` | `devices:control` 授权角色 1–4（`true`），角色 5 显式拒绝（`false`） |

### 3.2 迁移 049 — 命令能力表扩展与细粒度权限注册

**目标**：为命令级权限校验和动态表单提供元数据支撑。

**表结构变更 — `device_model_commands` 新增 12 列：**

| 字段名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| `config_domain` | VARCHAR(32) | — | 配置域：output / battery / service / parallel |
| `permission_code` | VARCHAR(64) | — | 细粒度权限码，映射至 `role_permissions` |
| `operation_kind` | VARCHAR(20) | `'persistent'` | 操作类型：persistent / transient |
| `requires_stopped` | BOOLEAN | `FALSE` | 是否要求设备处于停机状态 |
| `requires_bms_online` | BOOLEAN | `FALSE` | 是否要求 BMS 在线 |
| `requires_group_master` | BOOLEAN | `FALSE` | 是否要求设备为并机主机 |
| `confirmation_mode` | VARCHAR(20) | `'none'` | 确认模式：none / physical / risk_ack |
| `ttl_seconds` | INTEGER | — | 命令有效期（秒） |
| `cooldown_seconds` | INTEGER | — | 命令冷却期（秒） |
| `precondition_schema` | JSONB | `'{}'` | 前置条件校验 schema |
| `ui_schema` | JSONB | `'{}'` | 前端 UI 渲染 schema |
| `min_firmware` | VARCHAR(32) | — | 最低固件版本要求 |

**约束变更：**

- `risk_level` CHECK 约束由 `1..3` 扩展为 `0..4`（0=只读查询，4=出厂重置）

**数据回填：**

- 为 18 个已有命令回填 `config_domain` 和 `permission_code`，涵盖 AC 开关、功率限制、充放电限制、BMS 参数、SOC 窗口、强制充放电、重启、查询、出厂重置、参数写入、并机配置等

**索引：**

- `idx_dmc_model_domain` — `(model_id, config_domain) WHERE is_enabled = true`

**权限注册 — `role_permissions` 新增 8 个权限码：**

| 权限码 | (resource, action) | 授权角色 |
|--------|-------------------|---------|
| `device_control_basic` | (device_control, basic) | 1, 2, 3, 4, 5 |
| `device_control_disruptive` | (device_control, disruptive) | 1, 2, 3, 4 |
| `device_configure_battery` | (device_configure, battery) | 1, 4 |
| `device_configure_ac_input` | (device_configure, ac_input) | 1, 4 |
| `device_configure_parallel` | (device_configure, parallel) | 1, 2, 3, 4 |
| `device_service_diagnostics` | (device_service, diagnostics) | 1, 2, 3 |
| `device_service_factory` | (device_service, factory) | 无（仅超级管理员代码级旁路） |
| `ota_control` | (ota, control) | 1, 2, 3 |

## 5. 权限矩阵

### 5.1 角色定义

| 角色 ID | 角色名称 | 说明 |
|---------|---------|------|
| 1 | 原厂 | 超级管理员，拥有所有权限 |
| 2 | 总代理 | 区域代理，可管理下属设备与经销商 |
| 3 | 经销商 | 设备销售与服务商 |
| 4 | 安装商 | 现场安装与调试人员 |
| 5 | 用户 | 终端用户，仅可查看与基础控制 |

### 5.2 细粒度权限码 × 角色矩阵

| 权限码 | (resource, action) | r1 原厂 | r2 总代理 | r3 经销商 | r4 安装商 | r5 用户 |
|--------|-------------------|:-------:|:--------:|:--------:|:--------:|:------:|
| `device_control_basic` | (device_control, basic) | ✓ | ✓ | ✓ | ✓ | ✓ |
| `device_control_disruptive` | (device_control, disruptive) | ✓ | ✓ | ✓ | ✓ | — |
| `device_configure_battery` | (device_configure, battery) | ✓ | — | — | ✓ | — |
| `device_configure_ac_input` | (device_configure, ac_input) | ✓ | — | — | ✓ | — |
| `device_configure_parallel` | (device_configure, parallel) | ✓ | ✓ | ✓ | ✓ | — |
| `device_service_diagnostics` | (device_service, diagnostics) | ✓ | ✓ | ✓ | — | — |
| `device_service_factory` | (device_service, factory) | — | — | — | — | — |
| `ota_control` | (ota, control) | ✓ | ✓ | ✓ | — | — |

> **注**：`device_service_factory` 不授予任何角色，仅超级管理员通过代码级旁路执行。

### 5.3 权限码 → 命令映射

| 权限码 | 覆盖命令 |
|--------|---------|
| `device_control_basic` | `set_power_limit`、`set_soc_window`、`query_config`、`query_telemetry`、`get_params` |
| `device_control_disruptive` | `ac_on`、`ac_off`、`force_charge`、`force_discharge`、`restart`、`set_params`、`set_control`、`set_alarm`、`batch_config` |
| `device_configure_battery` | `set_charge_limit`、`set_discharge_limit`、`bms_set_charge_current`、`bms_set_discharge_current`、`bms_set_charge_voltage`、`bms_set_discharge_voltage`、`bms_set_enable` |
| `device_configure_parallel` | `parallel_enable`、`parallel_set_role`、`parallel_set_phase`、`parallel_disable` |
| `device_service_factory` | `reset`（出厂重置） |

### 5.4 9 步命令校验链

| 步骤 | 校验内容 | 拒绝码 |
|------|---------|--------|
| 1 | 身份、设备归属、细粒度 `permission_code` | `UNSUPPORTED_COMMAND` |
| 2 | 型号/固件是否支持命令（fail-closed） | `UNSUPPORTED_COMMAND` |
| 3 | 参数类型、单位、缩放、静态范围、枚举 | `INVALID_RANGE` |
| 4 | 多参数关系和互斥关系 | `INVALID_RELATION` |
| 5 | 设备在线、当前模式、停机/故障/忙状态 | `DEVICE_OFFLINE`、`DEVICE_BUSY`、`REQUIRES_STOPPED`、`ACTIVE_FAULT`、`UNSUPPORTED_IN_CURRENT_MODE` |
| 6 | BMS 在线及 CCL/DCL、电压、温度、SOC 动态限制 | `BMS_OFFLINE`、`BMS_LIMIT_EXCEEDED`、`TEMPERATURE_DERATING`、`BATTERY_PROFILE_MISMATCH` |
| 7 | 并机组级状态和拓扑限制 | `TOPOLOGY_CONFLICT`（当前为 TODO stub） |
| 8 | 风险确认、调试授权窗口、TTL 和冷却时间 | `AUTHORIZATION_EXPIRED`、`PHYSICAL_CONFIRMATION_REQUIRED` |
| 9 | 创建设备独立 `task_id` | — |

## 8. 部署注意事项

1. **迁移 048 必须在删除白名单代码部署前执行**
   - 迁移 048 将 6 个系统命令持久化到 `device_model_commands`。如果先部署 fail-closed 代码而迁移未执行，所有系统命令将被拒绝，导致设备不可控。

2. **所有在线设备的 `model_id` 必须在 fail-closed 上线前补全**
   - `fail-closed` 逻辑依赖 `device_model_commands` 查询命令是否允许。若设备的 `model_id` 为空或未匹配到型号记录，所有控制命令将被拒绝。

3. **迁移文件编号 048–052（非 026–030）**
   - 原设计文档中使用 026–030 编号，但该编号已被历史迁移占用，因此实际使用 048–052。

4. **数据库迁移通过 Docker Compose 自动执行**
   - 迁移文件放置于 `database/migrations/` 目录，容器启动时自动按序号执行。部署时确认 `deploy/docker-compose.yml` 中数据库服务挂载了正确的迁移目录。
