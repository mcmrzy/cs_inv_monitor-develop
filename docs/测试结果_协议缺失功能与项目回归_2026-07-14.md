## 4. 实际 HTTP 500 根因

### 4.1 并机状态

迁移 038 和 `schema.sql` 定义的 `device_parallel_state` 没有 `updated_at`，但 API UPSERT 同时在 INSERT 和 UPDATE 分支写入该列，因此主操作直接失败。

此外，`device_parallel_events` 定义的是 `machines_before/machines_after`，Go SQL 使用的却是 `old_state/new_state`。即使先补上 `updated_at`，拓扑事件写入仍会失败。

### 4.2 三相遥测

Device Server 当前发送：

- `voltage`、`current`、`active_power`、`line_voltage` 数组；
- `timestamp`；
- 不发送 `raw_envelope`。

API DTO 当前接收：

- `voltage_l1...voltage_l3` 等拆分字段；
- `event_time`；
- `raw_envelope`。

数组字段被 JSON 绑定器忽略，`event_time` 回落到服务器当前时间；`raw_envelope` 被显式以 NULL 写入，而表定义要求 `NOT NULL`，最终返回 500。即便只放宽 NULL，仍会写入一组错误的零值三相数据。

### 4.3 设备告警

`alarms` 的权威表结构只有 `alarm_level`、`fault_message` 等列，没有 `type`、`level`、`message`。当前 handler INSERT 使用了这三个不存在的列，所以告警主记录最先失败，后续事件和快照均不会执行。

## 5. 问题分级

### P0：发布阻断

1. **并机状态上报实际返回 500。** `device_parallel_state.updated_at` 不存在；事件表字段也与 Go SQL 不一致。
2. **三相遥测上报实际返回 500。** Device/API DTO 不一致，且 `raw_envelope` 显式 NULL 违反约束。
3. **设备告警上报实际返回 500。** `alarms.type/level/message` 不存在。
4. **告警事件结构与 Go 逻辑不一致。** 表中 `code/state` 为数值类型、`active_at` 为 NOT NULL、时间列为 `received_at`；Go 写入字符串状态，恢复事件传空 `active_at`，查询又使用不存在的 `created_at`。修复告警主表后，该链路仍会继续失败。
5. **Device Server unhealthy。** MQTT 证书 pin mismatch 持续循环，容器 8081 不可用，真实 MQTT → Device → API 链路无法建立。
6. **新增查询端点缺少资源归属校验。** 三个设备查询 handler 仅按请求路径中的 SN 查询，未在 SQL 中约束当前用户/租户对设备的 ownership，存在跨租户读取风险。此项为静态审查确认，未做越权破坏性测试。

### P1：高优先级

1. **迁移 036 后 `alarms` 丢失业务索引。** 实际数据库已确认新表仅有 `alarms_new_pkey`，设备、状态、时间索引留在 `alarms_old`，告警查询会退化为跨 chunk 扫描。
2. **迁移 036 down 有确定性语法错误。** 文件末尾在 PL/pgSQL 块外直接执行 `RAISE NOTICE`。
3. **迁移 036 回滚不合并新数据。** down 仅恢复迁移前的 `alarms_old`，迁移后新增数据仍留在备份超表，回滚后业务不可见；重复 down 还会与已有备份表冲突。
4. **并机/三相时间字段协议不一致。** Device Server 发 `timestamp`，API 接收 `reported_at/event_time`；修复数据库列后仍可能写入零时间或服务器时间。
5. **数据库集成测试断言与迁移 034 冲突。** 测试仍要求已被 034 删除的 `device_model_field` 存在，且未覆盖 038 新表的实际读写 SQL。
6. **Admin 单元测试未全绿。** `authStore` 两项失败，应在发布前恢复至 160/160。

### P2：需要排期

1. **告警快照保留策略不闭合。** `device_alarm_events` 保留一年，但快照表没有外键级联或 retention，长期会积累无法关联的孤儿快照。
2. **迁移 037 down 不可逆且可能误删。** PV 字段不恢复；up 对四条命令 `ON CONFLICT DO NOTHING`，down 却无条件删除，可能删除迁移前已有数据。
3. **`schema.sql` 与 `baseline_version=22` 边界不清晰。** schema 已包含 038 表，但缺少完整迁移后字段、策略和种子语义；“表存在”不能代表当前版本业务可用。
4. **前端 bundle 偏大。** 主 bundle 约 3.33 MB、gzip 后约 1.04 MB，建议路由拆包和依赖按需加载。

## 7. 结论与发布判定

本轮结果判定为 **失败 / 禁止发布**。

数据库已经迁移到 38，五张表也确实存在，但并机、三相和告警三个新增核心写入接口全部实测 HTTP 500；Device Server 同时因 MQTT 证书固定值不匹配处于 unhealthy。当前系统只能说明“容器和迁移框架部分可运行”，不能说明协议缺失功能已经打通。

建议修复顺序：

1. 先统一 038 表结构、API SQL 和 Device Server DTO，并为三个写入端点增加真实数据库集成测试。
2. 修复 `alarms` 主表列定义与 handler INSERT，随后验证告警主记录、事件、快照三层同时落库。
3. 修复迁移 036 的索引命名/清理顺序和 down 脚本，并执行一次 up/down 数据保全演练。
4. 配置真实 MQTT 证书摘要，使 Device Server 恢复 healthy，再执行端到端 MQTT 回归。
5. 补齐设备 ownership 校验、Admin 失败测试及 038 查询/保留策略测试。

满足以下条件后方可重新评估发布：三个内部写入接口均返回 2xx 且数据库数据正确、Device Server healthy、告警新表索引完整、Go 和 Admin 测试全部通过、越权查询验证被拒绝。
